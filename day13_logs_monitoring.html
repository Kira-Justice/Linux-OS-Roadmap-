1. Theory Section
syslog · journald · auditd · log levels · rotation

The Linux logging ecosystem

Linux has multiple overlapping logging systems capturing different layers of activity. Understanding each — what it captures, where it stores data, how to query it — is the foundation of incident response.

syslog / rsyslog
Traditional logging daemon. Routes messages from kernel and apps to files in /var/log/ by facility and severity rules.
journald (systemd)
Modern binary journal. Captures everything — kernel, services, boot. Queried with journalctl. Faster and more structured than text files.
auditd
Kernel-level security audit daemon. Logs syscalls, file accesses, user actions. Cannot be bypassed from userspace. Gold standard for forensic evidence.
Application logs
Each service writes its own — Apache in /var/log/apache2/, Nginx in /var/log/nginx/, MySQL in /var/log/mysql/.

Critical log files — know every one

/var/log/auth.log
Authentication events — SSH logins, sudo, su, PAM. First file checked in any incident. (Debian/Ubuntu)
/var/log/secure
Same as auth.log on RHEL/CentOS.
/var/log/syslog
General system messages — kernel, daemons. Catch-all for events with no dedicated log file.
/var/log/kern.log
Kernel-only messages — hardware errors, driver issues, kernel panics.
/var/log/dpkg.log
Package install/removal history with timestamps. Critical for attack timeline reconstruction.
/var/log/apt/history.log
apt command history. Every install, upgrade, removal — with the exact command and timestamp.
/var/log/cron
Cron job execution log — when each job ran and whether it succeeded.
/var/log/audit/audit.log
auditd kernel-level events. Read with ausearch and aureport.
/var/log/faillog
Binary file tracking failed login counts per user. Read with faillog command.

Syslog severity levels

0
emerg
System unusable. Kernel panic, total failure.
1
alert
Immediate action required. Data corruption risk.
2
crit
Critical conditions. Hardware and filesystem errors.
3
err
Error conditions. Service failures, config errors.
4
warning
Warning conditions. Unusual events, not yet failed.
5
notice
Normal but significant. Security-relevant actions.
6
info
Informational. Service starts, stops, connections.
7
debug
Verbose debug output. Disabled in most production configs.

Log rotation — how history is managed

Without rotation, logs fill disks. logrotate compresses and archives old logs on a schedule defined in /etc/logrotate.conf and /etc/logrotate.d/. This creates a sliding window — typically 7–52 weeks depending on config. During incident response, check if logs predate the event you're investigating — they may have rotated away.

cat /etc/logrotate.conf # global rotation config ls /etc/logrotate.d/ # per-app configs ls -lh /var/log/*.gz # compressed rotated logs zcat /var/log/auth.log.1.gz | grep "Failed" | head -10 # read compressed logs

2. Practical Section
journalctl · auditd · log pipelines · monitoring script

Exercise 1 — reading /var/log like a SOC analyst

ls -lh /var/log/ # all log files and sizes tail -f /var/log/auth.log # live auth stream tail -f /var/log/syslog # live system messages grep -c "Failed password" /var/log/auth.log # total failed login count grep "Accepted" /var/log/auth.log | tail -10 # recent successful logins grep "sudo" /var/log/auth.log | grep "COMMAND" | tail -10 # recent sudo commands grep "session opened" /var/log/auth.log | awk '{print $1,$2,$3,$11}' | tail -20

Exercise 2 — journalctl mastery

journalctl # all logs (q to quit, / to search) journalctl -f # live follow mode journalctl -b # current boot only journalctl -b -1 # previous boot (after crash or reboot) journalctl --since "2 hours ago" journalctl -p err # errors and above only journalctl -p warning -b # warnings from this boot journalctl -u ssh # SSH service logs only journalctl -u ssh -u cron -f # follow multiple services journalctl _UID=1000 # logs from a specific user UID journalctl --disk-usage # how much space the journal uses journalctl --vacuum-size=500M # trim journal to 500MB

Exercise 3 — auditd setup and rules

sudo apt install auditd audispd-plugins -y sudo systemctl enable --now auditd sudo auditctl -w /etc/passwd -p wa -k passwd_watch sudo auditctl -w /etc/shadow -p rwa -k shadow_watch sudo auditctl -w /etc/sudoers -p wa -k sudoers_watch sudo auditctl -w /tmp -p x -k tmp_exec sudo auditctl -a always,exit -F arch=b64 -S execve -k cmd_exec sudo auditctl -l # list all active rules # Persist rules across reboots: sudo tee /etc/audit/rules.d/security.rules << 'EOF' -w /etc/passwd -p wa -k passwd_watch -w /etc/shadow -p rwa -k shadow_watch -w /etc/sudoers -p wa -k sudoers_watch -w /etc/systemd/system -p wa -k systemd_persistence -w /etc/cron.d -p wa -k cron_persistence EOF sudo service auditd restart

Exercise 4 — ausearch and aureport

sudo ausearch -k passwd_watch # events matching key sudo ausearch -k shadow_watch -i # -i resolves UIDs to usernames sudo ausearch -m USER_LOGIN # all login events sudo ausearch -m USER_LOGIN --success yes # successful logins only sudo ausearch -m USER_LOGIN --success no # failed logins only sudo ausearch -x /bin/su # all su executions sudo ausearch --start today --end now # today's events only sudo aureport # full summary report sudo aureport --auth # auth events report sudo aureport --login # login events report sudo aureport --failed # failed events report

Exercise 5 — key log analysis pipelines

# Top 10 brute-force source IPs: grep "Failed password" /var/log/auth.log \ | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 # SSH login timeline — who, when, from where: grep "Accepted" /var/log/auth.log \ | awk '{print $1,$2,$3,"user:"$9,"from:"$11}' | tail -20 # All sudo commands with timestamp: grep "sudo.*COMMAND" /var/log/auth.log \ | awk '{print $1,$2,$3,$6,$(NF)}' | tail -20 # Package installs with timestamps: grep " install " /var/log/dpkg.log | tail -20 # New user account creations: grep "useradd\|new user" /var/log/auth.log | tail -10 # Service failures: journalctl -p err -b | grep "Failed to start\|failed"

Exercise 6 — build log-monitor.sh

cat > ~/security-lab/scripts/log-monitor.sh << 'EOF' #!/bin/bash LOG=/var/log/auth.log THRESHOLD=50 echo "=== AUTH SUMMARY $(date) ===" echo "Failed attempts : $(grep -c 'Failed password' $LOG 2>/dev/null)" echo "Successful logins: $(grep -c 'Accepted' $LOG 2>/dev/null)" echo "" echo "=== TOP BRUTE-FORCE IPs ===" grep "Failed password" $LOG 2>/dev/null \ | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 echo "" echo "=== ALERT: IPs EXCEEDING $THRESHOLD FAILURES ===" grep "Failed password" $LOG 2>/dev/null \ | awk '{print $11}' | sort | uniq -c | sort -rn \ | awk -v t=$THRESHOLD '$1>t {print "[!] "$2, "| attempts:", $1}' echo "" echo "=== SUCCESSFUL LOGINS (last 10) ===" grep "Accepted" $LOG 2>/dev/null | awk '{print $1,$2,$3,"from:"$11}' | tail -10 echo "" echo "=== RECENT SUDO COMMANDS ===" grep "COMMAND=" $LOG 2>/dev/null | tail -10 echo "" echo "=== RECENT PACKAGE INSTALLS ===" grep " install " /var/log/dpkg.log 2>/dev/null | tail -10 EOF chmod +x ~/security-lab/scripts/log-monitor.sh bash ~/security-lab/scripts/log-monitor.sh
3. Real-World Scenario
Log analysis reconstructs a 3-week breach in 10 minutes

A threat intelligence feed flags a company server as communicating with a known malicious address. The analyst opens a terminal — no SIEM, no dashboard, just log files and the tools from this roadmap.

Step 1 — establish the timeline

Checking /var/log/apt/history.log reveals an unexpected package installation 3 weeks ago at 2:47 AM on a Sunday — no maintenance window was scheduled. Two packages from an unrecognised third-party repository.

Step 2 — trace the initial access

grep "Accepted" /var/log/auth.log shows a successful SSH login from an unfamiliar IP at 2:31 AM — 16 minutes before the package install. grep "Failed password" from that same IP shows 847 failures in the 2 hours prior. Classic brute-force that succeeded against a weak service account password.

Step 3 — map all attacker actions

grep "COMMAND=" /var/log/auth.log | grep "service-account" shows every elevated command: creating a new user, installing software, writing to /etc/cron.d/. The entire 3-week attack chain is visible in auth.log alone.

Outcome and lessons

Full timeline reconstructed in under 10 minutes. Root cause: no log monitoring, no fail2ban, weak service account password — three issues from Days 5, 11, and 12 of this roadmap. The breach was a direct consequence of skipping those defensive steps.

4. Red Team Perspective
Log awareness · what leaves evidence · log retention

What actions leave log evidence and where

Understanding log coverage from the attacker perspective makes you a better defender — you know exactly what to monitor for.

# SSH connections: /var/log/auth.log → "Accepted publickey/password for USER from IP" # sudo usage: /var/log/auth.log → "sudo: USER : COMMAND=/path/to/cmd" # File access (if auditd rule watching the path): /var/log/audit/audit.log → SYSCALL and PATH records at kernel level # Software installed: /var/log/dpkg.log and /var/log/apt/history.log # Cron jobs executing: /var/log/cron or journalctl -u cron # All commands (if auditd execve rule is active): /var/log/audit/audit.log → type=EXECVE records

Checking for active monitoring before noisy actions

ps aux | grep -E "auditd|rsyslog|splunk|ossec|wazuh|filebeat|logstash" systemctl list-units --state=running | grep -iE "audit|syslog|monitor" ls /var/log/audit/ # is auditd active? sudo auditctl -l 2>/dev/null # what is being audited?

Log retention as intelligence for timing

ls -lh /var/log/auth.log* # how many rotations kept? cat /etc/logrotate.d/rsyslog # retention policy journalctl --disk-usage # journal size journalctl --list-boots | head -10 # how far back does journal go?

Shell history — the forgotten evidence trail

cat ~/.bash_history # commands run by this user history # current session commands # Shell history management (both sides need to know): export HISTFILE=/dev/null # discard history for session history -c # clear current session history # Note: clearing history is itself suspicious — defenders watch for this # in auditd logs as an indicator of evasion intent
5. Blue Team Perspective
Production auditd · log retention · remote forwarding · SIEM

Production auditd rules for any Linux server

sudo tee /etc/audit/rules.d/hardening.rules << 'EOF' # Identity and credential files -w /etc/passwd -p wa -k identity -w /etc/shadow -p wa -k identity -w /etc/sudoers -p wa -k identity -w /etc/group -p wa -k identity # SSH configuration and keys -w /etc/ssh/sshd_config -p wa -k sshd_config -w /root/.ssh -p wa -k root_ssh_keys # Persistence locations (Days 9 + 12) -w /etc/systemd/system -p wa -k systemd_persistence -w /etc/cron.d -p wa -k cron_persistence -w /var/spool/cron -p wa -k user_cron # All executed commands — verbose but comprehensive -a always,exit -F arch=b64 -S execve -k exec_log EOF sudo service auditd restart

Extend log retention beyond defaults

sudo tee /etc/systemd/journald.conf.d/retention.conf << 'EOF' [Journal] SystemMaxUse=2G MaxRetentionSec=90day EOF sudo systemctl restart systemd-journald # Extend syslog rotation to 52 weeks: sudo sed -i 's/rotate 4/rotate 52/' /etc/logrotate.conf

Forward logs to a remote server — tamper resistance

sudo tee /etc/rsyslog.d/50-remote.conf << 'EOF' # Forward all logs to central log server via TCP: *.* @@192.168.1.200:514 EOF sudo systemctl restart rsyslog # Why this matters: a root attacker CAN modify /var/log/auth.log locally. # They CANNOT reach your remote log server to modify what's already there. # Remote logging = tamper-resistant evidence trail.

10. Pro Tips & Mindset
How SOC analysts and incident responders think about logs

The first action in any incident investigation is establishing the timeline — not hunting for malware. Run ls -lt /var/log/ to see recently written logs. Check /var/log/apt/history.log for software installs, /var/log/auth.log for logins. Build the sequence of events before drawing any conclusions. Evidence before theory, always.

Always forward logs to a remote server in any environment that matters. A root attacker can modify local /var/log/auth.log but cannot reach your remote log server. Remote logs are your tamper-resistant evidence trail. A simple rsyslog forward to any second machine fundamentally changes your security posture.

Use journalctl -b -1 when a system has been rebooted during an incident. It shows all logs from the previous boot — including when services started and stopped — which pinpoints exactly when a persistence mechanism was triggered during the attack.

Common mistake: relying solely on /var/log/auth.log without cross-checking journald. If they diverge — journald shows an SSH event that auth.log doesn't — the text log may have been tampered with. Cross-reference both sources for any critical findings during a real investigation.

Common mistake: treating log monitoring as a one-time setup. Logs only help if someone reviews them. A weekly cron job running log-monitor.sh and writing output to a report file catches most attacks before they escalate. Unread logs are no better than no logs at all.

Pro mindset: the 2 hours spent setting up auditd, remote forwarding, and a monitoring script today will pay back the first time they catch an incident. Log infrastructure is cheap. Breach response is extremely expensive. The asymmetry is enormous — quantify it whenever making the case for logging investment to any organisation you work with.
