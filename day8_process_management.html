1. Theory Section
Processes · PID · states · signals · /proc

What is a process?

Every running program on Linux is a process — a program loaded into memory with its own resources (CPU time, memory, file handles). The OS gives each process a unique Process ID (PID). Understanding processes is fundamental to both offense and defense: attackers hide malware as processes, defenders hunt for anomalies in the process list.

PID
Process ID — unique integer assigned by the kernel. PID 1 is always init or systemd — the ancestor of all processes.
PPID
Parent Process ID — every process has a parent. The process tree shows who spawned what. Unexpected parents are a red flag.
UID/EUID
Real user and effective user. EUID is what matters for permissions — a SUID binary runs with EUID = owner, not the caller.
Nice value
CPU scheduling priority. Range -20 (highest) to 19 (lowest). Malware often runs at -20 to grab CPU; defenders look for unusual values.

Process states

Each process is always in one of these states — visible in the STAT column of ps aux:

R — Running S — Sleeping (interruptible) D — Disk sleep (uninterruptible) Z — Zombie (finished but not reaped) T — Stopped (paused) I — Idle kernel thread

Additional modifiers: s = session leader, l = multi-threaded, + = foreground process group, N = low priority (nice). Example: Ss = sleeping session leader.

Process signals

Signals are notifications sent to processes by the kernel or by users via kill. Despite the name, kill sends any signal — it does not only terminate processes.

1
SIGHUP
Hangup — often used to tell daemons to reload their configuration without restarting
2
SIGINT
Interrupt — what Ctrl+C sends to a foreground process. Can be caught and handled.
9
SIGKILL
Force kill — cannot be caught, blocked, or ignored. Instant termination. Use when SIGTERM fails.
15
SIGTERM
Graceful termination — default signal. Process can catch this and clean up before exiting.
18
SIGCONT
Continue a stopped process. Pair with SIGSTOP to pause/resume.
19
SIGSTOP
Pause a process — cannot be caught. Freezes execution without killing it.
11
SIGSEGV
Segmentation fault — process accessed memory it shouldn't. Often indicates a bug or exploit attempt.

The /proc filesystem — live process data

Every running process has a directory at /proc/[PID]/ containing live information readable as files — no special tool needed.

/proc/[PID]/cmdline
Full command that started the process, null-separated. What is this process actually running?
/proc/[PID]/exe
Symlink to the actual executable on disk. If it says (deleted), the binary was removed after launch — classic malware behaviour.
/proc/[PID]/maps
Memory map — what libraries and files are loaded. Useful for forensic analysis.
/proc/[PID]/net/tcp
Network connections opened by this process. Identifies which process owns which socket.
/proc/[PID]/environ
Environment variables at launch — may contain secrets, tokens, database URLs.
/proc/[PID]/fd/
All open file descriptors. What files, sockets, pipes does this process have open?

2. Practical Section
ps · top · htop · kill · /proc · jobs

Exercise 1 — ps: the process snapshot tool

ps aux # ALL processes, all users, extended format ps aux | head -5 # header + first 4 processes (understand columns) ps aux | grep root # only root-owned processes ps aux | grep -v root # processes NOT owned by root ps aux | sort -k3 -rn | head -10 # top 10 by CPU usage (column 3) ps aux | sort -k4 -rn | head -10 # top 10 by memory usage (column 4) ps -ef # full-format: shows PPID (parent PID) ps -ef --forest # ASCII tree showing parent-child relationships ps -eo pid,ppid,user,stat,comm # custom columns: PID, PPID, user, state, command

Understanding ps aux columns

USER PID %CPU %MEM VSZ RSS TTY STAT START TIME COMMAND root 1 0.0 0.1 168736 9876 ? Ss 08:01 0:02 /sbin/init www-data 812 0.3 1.2 456720 25432 ? S 08:05 0:15 /usr/sbin/apache2 # VSZ = virtual memory size (total allocated) # RSS = resident set size (actually in RAM right now) # TTY = terminal ? = no terminal (background daemon) # STAT = process state (S=sleep, R=run, Z=zombie, T=stopped)

Exercise 2 — top: live process monitor

top # launch top (q to quit) top -u www-data # show only www-data processes top -p 1234 # monitor specific PID only top -b -n 1 # batch mode: one snapshot, no interactive UI top -b -n 1 | head -20 # good for scripting # Interactive keys inside top: # k = kill a process (enter PID then signal) # r = renice (change priority) # M = sort by memory # P = sort by CPU # u = filter by user # 1 = show individual CPU cores # q = quit

Exercise 3 — htop: the better top

sudo apt install htop -y # install if needed htop # launch — arrow keys to navigate htop -u root # only root processes htop -p 1,2,3 # specific PIDs only # htop interactive: # F3 or / = search for process by name # F5 = tree view (shows parent-child) # F6 = choose sort column # F9 = send signal to selected process # F10 = quit # Space = tag a process (select multiple)

Exercise 4 — sending signals with kill and killall

sleep 9999 & # start a background process to practice on ps aux | grep sleep # find its PID kill [PID] # sends SIGTERM (15) — graceful kill -9 [PID] # sends SIGKILL — force (use when SIGTERM fails) kill -l # list all signal names and numbers killall sleep # kill all processes named "sleep" killall -9 firefox # force-kill all firefox processes pkill -u www-data # kill all processes owned by www-data pgrep sshd # get PID(s) of all sshd processes without grep

Exercise 5 — background jobs and process control

sleep 300 & # run sleep in background (&) sleep 400 & # another one jobs # list background jobs in current shell fg %1 # bring job 1 to foreground Ctrl+Z # suspend foreground process (sends SIGSTOP) bg %1 # resume suspended job in background disown %1 # detach job so it survives shell exit nohup long-command & # run immune to hangup signal (survives logout)

Exercise 6 — /proc for forensic process inspection

ls /proc | head -20 # all process PIDs as directories cat /proc/$$/cmdline | tr '\0' ' ' # current shell's command line cat /proc/1/cmdline | tr '\0' ' ' # PID 1 command line (init/systemd) ls -la /proc/$$/exe # what binary is running as current shell ls -la /proc/[PID]/exe # binary behind any process cat /proc/[PID]/environ | tr '\0' '\n' # environment variables of a process ls /proc/[PID]/fd | wc -l # count open file descriptors cat /proc/meminfo # system memory breakdown cat /proc/loadavg # 1/5/15-minute load averages

Exercise 7 — niceness and priority

nice -n 10 sleep 100 & # start with low priority (nice 10) ps -o pid,ni,comm -p [PID] # check nice value of a process sudo renice -n -5 [PID] # raise priority (needs root for negative values) sudo renice -n 19 [PID] # lowest priority — system scraps
3. Real-World Scenario
Malicious process discovered via anomaly in ps output

A sysadmin notices a production web server is running unusually hot — CPU is at 94% but no deployments have been made. They investigate using only the tools from today's lesson.

Step 1 — check what's consuming CPU

They run top and immediately see a process called kworker/2:1 consuming 87% CPU. Legitimate kernel workers never use that much CPU consistently — this is a red flag. The process is owned by www-data, not the kernel.

Step 2 — inspect the suspicious process

They note the PID (say, 3847) and run ls -la /proc/3847/exe. Instead of pointing to a kernel binary, it shows /tmp/.k/kworker (deleted) — the binary was launched from /tmp and then deleted from disk to hide it. Classic cryptominer behaviour.

Step 3 — check parent and network activity

ps -ef | grep 3847 shows its parent is PID 812, which is the Apache www-data process — confirming a web shell or RCE was used to launch it. cat /proc/3847/net/tcp shows an active outbound connection to an external mining pool IP.

Step 4 — find persistence

find /tmp /var/tmp -name ".*" -type f 2>/dev/null finds a hidden dropper script. crontab -l -u www-data reveals a cron job re-launching the miner every 5 minutes — which is why killing it hasn't worked.

Resolution

Remove the cron entry first, then kill the process. Patch the web vulnerability that allowed RCE. A /proc/[PID]/exe (deleted) entry is now one of the analyst's standard checks during any incident — learned directly from this scenario.

4. Red Team Perspective
Process injection · Hiding in plain sight · /proc abuse

Process enumeration after foothold

A pentester inspects the process list immediately after gaining a shell — it reveals what services are running, what users are active, and what software versions might be vulnerable.

ps aux # full process list ps aux | grep -v "\[" | awk '$1!="root"' # non-root, non-kernel processes ps -ef --forest # tree: spot unusual parent-child pairs ps aux | sort -k3 -rn | head -5 # what's consuming the most CPU? ps aux | awk '$1=="root" && $3>10' # root processes eating CPU (unusual)

Spotting defensive tools in the process list

Before running noisy tools, attackers check if any security monitoring is active:

ps aux | grep -iE "splunk|osquery|wazuh|falco|auditd|sysmon|crowdstrike|sentinel" ps aux | grep -iE "antivirus|clamav|sophos|carbon.?black" ls /proc/*/exe 2>/dev/null | xargs ls -la 2>/dev/null | grep -iE "splunk|osquery"

Extracting credentials from process environment variables

for pid in $(ls /proc | grep -E '^[0-9]+$'); do cat /proc/$pid/environ 2>/dev/null | tr '\0' '\n' \ | grep -iE "password|secret|token|key|api" \ && echo " [PID $pid]" done

Database URLs, API keys, and cloud credentials are frequently passed to processes as environment variables — and they live in /proc/[PID]/environ readable by root or the process owner.

Process name masquerading

Attackers rename malicious processes to blend in with legitimate system processes. Names like kworker, kthreadd, bash, or sshd are commonly used. The giveaway is checking /proc/[PID]/exe — a process named kworker should point to a kernel path, not /tmp.

ls -la /proc/*/exe 2>/dev/null | grep -v "kernel\|/usr\|/bin\|/sbin\|/lib" ls -la /proc/*/exe 2>/dev/null | grep "deleted" # running from deleted binary
5. Blue Team Perspective
Process baselining · Anomaly detection · Incident triage

Establish a process baseline

Know what normal looks like before an incident. On a clean server, capture the expected process list and compare against it during investigations.

ps aux --sort=pid > /root/process-baseline.txt # save clean baseline diff /root/process-baseline.txt <(ps aux --sort=pid) # compare current vs baseline ps aux | awk '{print $1,$11}' | sort -u > /root/proc-users.txt # user:process pairs

Hunt for malicious process indicators

ls -la /proc/*/exe 2>/dev/null | grep "(deleted)" # running from deleted binary ls -la /proc/*/exe 2>/dev/null | grep "/tmp\|/dev/shm\|/var/tmp" # ran from writable dir ps aux | awk '$3 > 80 {print}' # any process >80% CPU ps aux | awk '$11 ~ /^\[/ {next} $1!="root" && $11 ~ /^[^/]/ {print}' # processes whose command doesn't start with / (may be masquerading)

Zombie and orphan process detection

ps aux | awk '$8 ~ /Z/ {print "ZOMBIE:", $0}' # all zombie processes ps aux | awk '$8 ~ /D/ {print "UNINTERRUPTIBLE:", $0}' # D-state (disk wait — may indicate I/O attack) ps -eo pid,ppid,stat,comm | awk '$3 ~ /Z/'

Monitor process creation with auditd

auditctl -a always,exit -F arch=b64 -S execve -k process_exec ausearch -k process_exec | tail -30 # recent process executions ausearch -k process_exec | grep "www-data" # processes spawned by web user

Respond to malicious process

kill -9 [PID] # terminate immediately cat /proc/[PID]/cmdline # capture command before killing (evidence) cp /proc/[PID]/exe /root/malware-sample # copy the binary before it's killed ls /proc/[PID]/fd # what files/sockets did it have open? cat /proc/[PID]/maps # memory map for forensic analysis

10. Pro Tips & Mindset
How professionals think about processes

Always check /proc/[PID]/exe before killing a suspicious process — and copy the binary first if you can (cp /proc/[PID]/exe /root/evidence-sample). Once you kill the process, the /proc/[PID]/ directory disappears permanently. Evidence first, containment second.

Processes named to mimic kernel threads (like kworker, kthreadd, ksoftirqd) are a common evasion technique. Real kernel threads appear in square brackets in ps aux output: [kworker/0:0]. If you see one without brackets, it's a userspace process masquerading. Investigate immediately.

Use ps -ef --forest not just ps aux. The tree view immediately reveals abnormal parent-child relationships — a web server process spawning a bash shell is a major red flag (possible web shell execution), invisible in the flat ps aux view.

Common mistake: using kill -9 immediately without trying kill -15 (SIGTERM) first. SIGKILL does not allow cleanup — databases may corrupt, log files may be left incomplete, and important forensic state is lost. Always try SIGTERM first in non-emergency situations.

Common mistake: confusing process priority. Lower nice values mean higher priority — nice -n -20 is the highest priority, not the lowest. The name is counterintuitive: being "nice" means yielding to others (higher number = lower priority = more polite to other processes).

Pro mindset: in incident response, the process list is your first window into what happened. Before touching anything else, capture it with ps aux > /tmp/ps-capture.txt and timestamp it. This snapshot becomes evidence. Systems change every second — the process that was running malware may self-terminate in minutes.
