1. Theory Section
apt · dpkg · repos · GPG signing · supply chain

How Linux package management works

A package manager is a system for installing, upgrading, configuring, and removing software. It handles dependencies automatically and — critically for security — verifies the integrity and authenticity of packages using cryptographic signatures. Understanding this pipeline is essential because every unpatched package is a potential vulnerability, and every untrusted repository is a potential supply chain attack.

apt
High-level package manager for Debian/Ubuntu. Resolves dependencies, downloads, and installs packages. The tool you use daily.
dpkg
Low-level package tool — installs, removes, and queries .deb files directly. apt calls dpkg under the hood.
apt-get / apt-cache
Older CLI variants of apt. Still widely used in scripts. apt-cache search is fast for finding packages.
snap / flatpak
Containerised package formats. Sandboxed but have their own security model — snap packages run with confinement profiles.

The package lifecycle — from repo to disk

Repository
A server hosting .deb packages and metadata. Defined in /etc/apt/sources.list and /etc/apt/sources.list.d/. Each repo is signed with a GPG key.
GPG signature
Every package and repo metadata is cryptographically signed. apt verifies the signature before installing anything. A tampered package will be rejected.
apt update
Downloads fresh package lists (metadata) from all repos. Does NOT install anything — just refreshes what's available.
apt upgrade
Installs available updates for all installed packages. Run this regularly — unpatched packages are a primary attack vector.
Package cache
Downloaded .deb files are cached in /var/cache/apt/archives/. Can be cleared with apt clean.

Supply chain security — why this matters for security professionals

Package managers are a trust chain. You trust the OS vendor, who trusts the package maintainer, who trusts the upstream developer. Each link is an attack surface:

  • Malicious repository: An attacker adds a fake repo to /etc/apt/sources.list.d/ that serves a backdoored version of a common tool.
  • Typosquatting: A malicious package named openssh-server vs openssl-server — one character difference, same install command.
  • Compromised upstream: The legitimate package source is compromised — the attacker modifies source code before the maintainer packages it.
  • GPG key theft: An attacker who steals a maintainer's signing key can sign malicious packages that apt accepts without warning.

Always verify repository sources, GPG key fingerprints, and checksums of binaries installed outside the package manager.

2. Practical Section
apt · dpkg · repos · security tools installation

Exercise 1 — apt fundamentals

sudo apt update # refresh package lists (always run first) sudo apt upgrade # install all available updates sudo apt full-upgrade # upgrade + handle dependency changes sudo apt install nmap -y # install a package sudo apt install nmap wireshark netcat-openbsd -y # install multiple sudo apt remove nmap # remove package (keep config files) sudo apt purge nmap # remove package AND config files sudo apt autoremove # remove orphaned dependencies sudo apt clean # delete cached .deb files apt show nmap # detailed info about a package apt list --installed # all installed packages apt list --installed | grep -c "" # count installed packages

Exercise 2 — apt search and investigation

apt search nmap # search by name/description apt-cache search port scanner # keyword search apt-cache show nmap # version, size, description, dependencies apt-cache depends nmap # what does nmap depend on? apt-cache rdepends nmap # what depends on nmap? apt-cache policy nmap # installed vs available versions + repo source dpkg -l | grep nmap # confirm installation and version dpkg -L nmap # list all files installed by a package dpkg -S /usr/bin/nmap # which package owns this file?

Exercise 3 — dpkg for .deb files

dpkg -i package.deb # install a local .deb file dpkg -r package-name # remove package dpkg -l # list all installed packages dpkg -l | grep "^ii" # only correctly installed packages dpkg -l | grep "^rc" # packages removed but config remains dpkg --get-selections # full package selection state

Exercise 4 — repository management and GPG

cat /etc/apt/sources.list # main repo sources ls /etc/apt/sources.list.d/ # additional repos (audit these regularly) apt-key list # trusted GPG keys (deprecated but shows existing) ls /etc/apt/trusted.gpg.d/ # modern location for trusted repo keys sudo apt-key fingerprint # verify key fingerprints # Add an external repo safely (example: GitHub CLI): curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg \ | sudo dd of=/usr/share/keyrings/githubcli-archive-keyring.gpg echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/githubcli-archive-keyring.gpg] \ https://cli.github.com/packages stable main" \ | sudo tee /etc/apt/sources.list.d/github-cli.list

Exercise 5 — essential security tools installation

sudo apt update # Network analysis and scanning: sudo apt install -y nmap wireshark tshark netcat-openbsd tcpdump # Vulnerability and enumeration: sudo apt install -y nikto dirb gobuster hydra # Forensics and analysis: sudo apt install -y foremost binwalk strings hexedit # Password and hash tools: sudo apt install -y john hashcat # System monitoring and audit: sudo apt install -y htop iotop nethogs fail2ban auditd aide lynis # Useful utilities every security person needs: sudo apt install -y curl wget git vim tree jq python3-pip net-tools dnsutils

Exercise 6 — verify installed tools and check for updates

which nmap wireshark netcat john # verify tools are in PATH nmap --version # version checks for key tools john --version lynis --version apt list --upgradable # what has available updates? sudo apt list --upgradable 2>/dev/null | grep -v "^Listing" # clean output # Check for security updates specifically: sudo apt-get --just-print upgrade 2>&1 | grep "^Inst" | grep -i security

Exercise 7 — lynis: automated system audit

sudo apt install lynis -y sudo lynis audit system # full system security audit (takes ~2 min) sudo lynis audit system --quick # faster version # After the audit: cat /var/log/lynis.log # full detailed log cat /var/log/lynis-report.dat # machine-readable findings grep "warning\|suggestion" /var/log/lynis-report.dat | head -20 # Lynis gives you a hardening index score and prioritised recommendations # This is exactly what security auditors run on new systems
3. Real-World Scenario
Malicious third-party repository backdoors 40,000 servers

A popular Linux tutorial blog publishes a guide for setting up a development environment. The guide instructs readers to add a third-party apt repository and install a toolchain. The repository is later compromised.

Step 1 — the malicious repo is introduced

An attacker compromises the blog's repository server and replaces legitimate package files with backdoored versions. The packages are still signed with the same GPG key (the attacker also stole the private key). apt considers the packages fully trusted — the signature is valid.

Step 2 — mass installation via trusted instructions

Thousands of developers and sysadmins following the guide run sudo apt update && sudo apt install toolchain. They install the backdoored package silently. No warnings appear — the signature is valid, the package name is correct.

Step 3 — persistence across 40,000 systems

The backdoor establishes a systemd service on each system (sound familiar from Day 9?) that phones home. By the time the compromise is discovered, tens of thousands of servers have been affected for weeks.

Step 4 — detection is delayed

Because the installation was legitimate-looking (correct package name, valid signature, expected behavior), no automated security tools flagged it. Detection came from a researcher who noticed anomalous outbound connections in network traffic analysis.

Defender lessons

Audit /etc/apt/sources.list.d/ regularly — every third-party repo is a trust extension. Verify GPG key fingerprints against official sources. Use dpkg -l to audit installed packages. Monitor outbound network connections for unexpected destinations. Tools like lynis flag unrecognised repos as a finding.

4. Red Team Perspective
Package enumeration · software versions · living off the land

Installed software enumeration after foothold

The package list is a goldmine for finding exploitable software. Every installed package with a known CVE is a potential privilege escalation or lateral movement path.

dpkg -l # full installed package list dpkg -l | awk '{print $2,$3}' | grep -v "^Name\|^+-" | sort # clean package:version list for vulnerability lookup dpkg -l | grep -iE "openssh|apache|nginx|mysql|postgres|php|python|perl|ruby" # look for network-facing services with version numbers cat /etc/apt/sources.list /etc/apt/sources.list.d/* 2>/dev/null # what repos is this system pulling from? any unusual third-party repos?

Version fingerprinting for CVE lookup

dpkg -l openssh-server # SSH server version dpkg -l apache2 2>/dev/null # Apache version dpkg -l linux-image-* | grep "^ii" # kernel version from packages uname -r # current running kernel version # Cross-reference versions at: # https://cve.mitre.org/cve/search_cve_list.html # https://nvd.nist.gov/vuln/search

Tools found on the system — LOLBins and more

which python3 perl ruby php curl wget nc socat ncat 2>/dev/null # what scripting runtimes and network tools are available? dpkg -l | grep -iE "netcat|socat|ncat|nc" 2>/dev/null # network redirection tools — useful for shells and tunnelling find /usr/bin /usr/sbin /bin /sbin -perm -u+x -type f | xargs dpkg -S 2>/dev/null | grep -v "^dpkg" # map every executable to its owning package — spot unpackaged binaries

Identifying unpackaged binaries (possible implants)

find /usr/local/bin /usr/local/sbin /opt -type f -executable 2>/dev/null # unpackaged locations — not managed by apt, not tracked find /usr/bin /bin -type f | while read f; do dpkg -S "$f" &>/dev/null || echo "UNPACKAGED: $f" done # any executable in standard bin dirs not owned by a package is suspicious
5. Blue Team Perspective
Patch management · repo audit · lynis · unattended upgrades

Automate security updates

sudo apt install unattended-upgrades -y sudo dpkg-reconfigure --priority=low unattended-upgrades # configure automatic security-only updates cat /etc/apt/apt.conf.d/50unattended-upgrades # review what's configured for auto-upgrade sudo unattended-upgrades --dry-run --debug # test without applying

Audit installed packages and repositories

cat /etc/apt/sources.list ls -la /etc/apt/sources.list.d/ # audit all third-party repos ls /etc/apt/trusted.gpg.d/ # trusted signing keys dpkg -l | grep "^rc" # packages removed but config remains — may contain credentials dpkg --get-selections | grep deinstall # packages marked for removal but not cleaned find /usr/local/bin /opt -type f -executable -newer /etc/apt/sources.list 2>/dev/null # executables installed after the repo config was last changed — anomaly indicator

Run lynis for automated security auditing

sudo lynis audit system # After completion, check the hardening index and focus on: grep "warning" /var/log/lynis-report.dat # security warnings grep "suggestion" /var/log/lynis-report.dat | head -20 # hardening suggestions grep "hardening_index" /var/log/lynis-report.dat # overall score (aim for 80+)

Check for vulnerable packages

sudo apt-get --just-print upgrade 2>&1 | grep "^Inst" | grep -i security # packages with available security updates sudo apt list --upgradable 2>/dev/null sudo apt upgrade --dry-run # what would upgrade change? debsecan 2>/dev/null || sudo apt install debsecan -y && debsecan # Debian Security Analyzer — lists CVEs affecting installed packages

Add package audit to recon.sh

banner "PACKAGE AUDIT" echo "--- Third-party repositories ---" cat /etc/apt/sources.list.d/* 2>/dev/null echo "--- Packages with security updates available ---" apt-get --just-print upgrade 2>&1 | grep "^Inst" | grep -i security | head -20 echo "--- Unpackaged executables in /usr/local ---" find /usr/local/bin /usr/local/sbin -type f -executable 2>/dev/null
Essential Security Tools Reference
Install · use · know why each matters

Network and scanning

nmap
Port scanner and service fingerprinter. The most widely used recon tool. nmap -sV -sC target is the standard starting scan. core tool
wireshark
GUI packet capture and analysis. See every byte of network traffic. Essential for understanding protocols and detecting anomalies. core tool
tshark
CLI version of Wireshark. Scriptable packet capture and analysis. tshark -i eth0 -w capture.pcap
tcpdump
Lightweight CLI packet capture. Always available on servers without Wireshark. tcpdump -i eth0 -w file.pcap
netcat (nc)
The Swiss Army knife of networking. Port scanning, banner grabbing, file transfer, shells. nc -lvnp 4444 listens for connections. core tool

Web application and enumeration

nikto
Web server scanner. Detects misconfigurations, outdated software, dangerous files. nikto -h http://target
gobuster
Directory and file brute-forcer. Find hidden paths on web servers. gobuster dir -u http://target -w /usr/share/wordlists/dirb/common.txt
curl / wget
Already installed. HTTP requests, header inspection, file downloading. curl -I url for headers. already installed

Password and hash analysis

john (John the Ripper)
Password hash cracker. Supports hundreds of hash formats. john --wordlist=rockyou.txt hashes.txt use legally and ethically only
hashcat
GPU-accelerated hash cracker. Much faster than John on supported hardware. hashcat -m 1800 hashes.txt wordlist.txt use legally and ethically only

System hardening and audit

lynis
Automated security auditing tool. Scans the entire system and gives a hardening index score with prioritised recommendations. Run on every new server. run now
auditd
Kernel-level audit daemon. Logs file accesses, system calls, user actions. Already covered in Week 1 — now you know how to install it on new systems. already covered
fail2ban
Intrusion prevention. Bans IPs after repeated failed auth attempts. Covered Day 11. already covered
aide
File integrity monitor. Creates a database of filesystem state — alerts when files change unexpectedly. aide --init then aide --check

Forensics and malware analysis

strings
Extract printable strings from any binary or file. First thing to run on a suspicious file. strings /tmp/suspicious-binary | head -50
binwalk
Firmware and binary analysis. Identify embedded files and file systems. Used in CTF challenges constantly.
foremost
File carving tool. Recover files from disk images or raw data based on file headers.

10. Pro Tips & Mindset
Package hygiene and tool habits of professionals

Always run sudo apt update before installing anything. Without it, apt works from a potentially stale package list and may install an older version — or fail entirely on a new repo. Make it a reflex: update then install, always in that order.

Run lynis audit system on every new server you set up or take responsibility for. The 2 minutes it takes produces a prioritised hardening checklist that would take hours to compile manually. Your hardening index score is also a useful metric to track over time — it should go up as you apply recommendations.

Use dpkg -S /path/to/binary to answer "what package owns this file?" any time you encounter an unexpected executable. If the answer is "dpkg: no path found matching pattern" — that binary is not managed by apt and deserves investigation. This is one of the fastest ways to spot implants or unmanaged software on a system.

Never pipe a shell script directly from curl into bash without reading it first — curl https://example.com/install.sh | sudo bash is a common installation pattern but it executes whatever the server returns, bypassing every security check. Download the script first, inspect it, then run it: curl -o install.sh URL && less install.sh && bash install.sh.

Don't install every security tool on every system. Production servers should have the minimum software necessary — each additional package is additional attack surface. Security tools like nmap and nikto belong on your dedicated security workstation or a dedicated scanning VM, not on a production database server.

Pro mindset: think of your system's package list as a security inventory. Every installed package is a responsibility — you are responsible for keeping it updated, knowing what it does, and removing it when it's no longer needed. apt autoremove after removing a package cleans orphaned dependencies. A lean, well-maintained package list is meaningfully more secure than a bloated one.
