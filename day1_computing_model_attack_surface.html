1. Theory Section
Core concepts · Definitions · How it works

The Computing Model (OSI-inspired stack)

Every system you will ever attack or defend is made of layered components. Understanding this stack is the foundation of all cybersecurity thinking.

Hardware (CPU, RAM, Disk, NIC) ↓ OS Kernel (Linux, Windows — manages resources) ↓ System Libraries (libc, OpenSSL, etc.) ↓ Applications (web servers, databases, browsers) ↓ Network (Ethernet → IP → TCP → HTTP/S) ↓ User / Human (the biggest attack surface of all)

What is an attack surface?

The attack surface is the total sum of different points (called attack vectors) where an unauthorized user can try to enter or extract data. The larger the surface, the more risk.

Attack Vector
A specific path or method an attacker uses to reach a target
Attack Surface
ALL possible vectors combined — every open port, service, user
Threat
Any potential danger that could exploit a vulnerability
Vulnerability
A weakness in a system that can be exploited by a threat

The CIA Triad — the core security model

  • Confidentiality — only authorized users see data (encryption, access control)
  • Integrity — data is not tampered with (hashes, digital signatures)
  • Availability — systems are up when needed (DDoS prevention, backups)

Key first-principles concepts

  • Least privilege: give users/processes only the minimum permissions needed
  • Defense in depth: multiple layers of security, not just one wall
  • Assume breach: design as if attackers are already inside
  • Zero trust: never trust, always verify — even internal traffic

2. Practical Section
Hands-on commands · Step-by-step exercises

Exercise 1 — map your own attack surface

Open a Linux terminal (or use TryHackMe's AttackBox). Run these commands and observe what's exposed:

uname -a # OS version + kernel hostname # machine name whoami # current user id # user ID, group memberships cat /etc/os-release # distro details

Exercise 2 — see what's listening on the network

ss -tuln # open TCP/UDP ports netstat -tuln # alternative (older systems) cat /etc/services # what services map to which ports

Exercise 3 — list running processes (potential attack surface)

ps aux # all running processes ps aux | grep root # root-owned processes (high risk) top # live process viewer

Exercise 4 — check installed software (version = vulnerability)

dpkg -l # Debian/Ubuntu installed packages rpm -qa # RHEL/CentOS installed packages which nc nmap curl wget # check for common tools present

Exercise 5 — map network connections

ip addr # all network interfaces + IPs ip route # routing table cat /etc/hosts # local DNS mappings cat /etc/resolv.conf # DNS server config

After each command, ask yourself: "What could an attacker learn from this output?" This mindset shift is the core of security thinking.

3. Real-World Scenario
Attack & defense story based on today's topic

Scenario: A fintech startup launches a new payment API. The developer left SSH (port 22) open on the production server with default credentials, and exposed a debug endpoint on port 8080.

Attacker reconnaissance

The attacker runs a port scan using Shodan (an internet search engine for devices) and finds the exposed server. They see port 22 and 8080 open.

Exploitation

They attempt admin:admin and root:password on SSH. It succeeds. They now have shell access to a production payment server.

Impact

The debug endpoint at :8080/debug/env leaks database credentials. The attacker exfiltrates 40,000 customer records.

What should have happened

Firewall rules limiting SSH to known IPs, key-based SSH auth only, no debug endpoints in production, regular port scanning of your own infrastructure.

Lesson: The attack surface was unnecessarily large. Every open port is a door. Every service is a potential vulnerability.

4. Red Team Perspective
How attackers think and exploit this concept

Attacker mindset: enumerate everything

Before exploiting anything, attackers map the attack surface completely. The methodology is: Reconnaissance → Scanning → Enumeration → Exploitation.

What attackers look for on a computing model

  • Hardware layer: physical access, USB attacks, BIOS/UEFI firmware exploits
  • OS layer: outdated kernel, misconfigurations, SUID binaries, cron jobs
  • Application layer: unpatched software, default credentials, debug modes left on
  • Network layer: open ports, unencrypted protocols (Telnet, FTP, HTTP), weak TLS
  • Human layer: phishing, social engineering, password reuse — most successful vector

Red team tools introduced (learn these over time)

nmap -sV target_ip # port scan + service version detection shodan.io # search internet-exposed devices theHarvester # OSINT: emails, domains, IPs whois target.com # domain ownership info

Key red team principle

Attackers don't need to exploit every vulnerability — just one entry point. That's why reducing attack surface is more effective than trying to patch every flaw.

5. Blue Team Perspective
Detect · Prevent · Respond

Reduce the attack surface (Prevention)

  • Close all ports not in use — run ss -tuln regularly and close unexpected listeners
  • Remove unused software — every extra package is potential vulnerability
  • Enforce least privilege — users and services get only what they need
  • Disable default accounts and change default credentials immediately
  • Use firewalls (ufw, iptables) to allow only necessary traffic

Detect anomalies (Detection)

last # recent login history lastb # failed login attempts cat /var/log/auth.log # SSH and auth logs journalctl -xe # systemd logs (recent events) ausearch -m USER_LOGIN # auditd login events

Respond (Incident Response basics)

  • Isolate the affected system from the network immediately
  • Preserve logs before they are overwritten or deleted
  • Identify the entry point — which service, which credential, which port
  • Patch the vulnerability, then restore from clean backup

Blue team principle

You can't protect what you don't know exists. Asset inventory and regular surface mapping are the foundation of all defense.

10. Pro Tips & Mindset
Advice · Common mistakes · How pros think

Think in layers, always. When you see any system, instinctively ask: what layer am I looking at, and what could go wrong here?

Curiosity over memorization. Don't memorize commands. Understand WHY each command exists. Google and man pages exist for syntax. Thinking doesn't.

Document everything. Professionals keep notes on every system they touch. Start this habit on Day 1. Your notes become your second brain.

Common mistake: Skipping theory and jumping to tools. Tools change. Concepts don't. Nmap is useless if you don't understand what a port is.

Common mistake: Practicing only on your own machine. Use TryHackMe/HTB legal environments. Never scan systems you don't own.

Common mistake: Treating cybersecurity as only technical. The human layer is the most exploited attack surface. Psychology matters as much as code.

Pro mindset: Always ask "what's the impact?" not just "is this exploitable?" Severity = likelihood × impact. This is how real security reports are written.
