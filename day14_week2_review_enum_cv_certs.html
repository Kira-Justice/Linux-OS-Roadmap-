Complete enum.sh script

#!/bin/bash # ============================================================ # enum.sh — Week 2 Linux Enumeration Script # Author : [Your Name] # Purpose : Network, process, service and log enumeration # Usage : sudo bash enum.sh [--output /path/report.txt] # Builds on recon.sh from Week 1 # ============================================================ OUTPUT="${2:-$HOME/security-lab/evidence/enum-$(date +%Y%m%d-%H%M%S).txt}" mkdir -p "$(dirname "$OUTPUT")" exec > >(tee "$OUTPUT") 2>&1 RED='\033[0;31m'; YEL='\033[0;33m'; GRN='\033[0;32m' CYN='\033[0;36m'; NC='\033[0m'; BOLD='\033[1m' banner() { echo -e "\n${BOLD}${CYN}========== $1 ==========${NC}"; } warn() { echo -e "${YEL}[!] $1${NC}"; } alert() { echo -e "${RED}[ALERT] $1${NC}"; } ok() { echo -e "${GRN}[OK] $1${NC}"; } echo -e "${BOLD}enum.sh | $(date) | $(hostname)${NC}" echo "Report: $OUTPUT" # ------ SECTION 1: PROCESS AUDIT (Day 8) ------ banner "PROCESS AUDIT" echo "--- Top 10 CPU-consuming processes ---" ps aux --sort=-%cpu | head -11 echo "" echo "--- Processes running from deleted binaries (malware indicator) ---" ls -la /proc/*/exe 2>/dev/null | grep "(deleted)" | while read line; do alert "DELETED BINARY: $line" done echo "" echo "--- Processes running from writable locations ---" ls -la /proc/*/exe 2>/dev/null | grep -E "/tmp|/dev/shm|/var/tmp" | while read line; do alert "SUSPICIOUS LOCATION: $line" done echo "" echo "--- Zombie processes ---" ps aux | awk '$8 ~ /Z/ {print "[ZOMBIE]", $0}' # ------ SECTION 2: PERSISTENCE AUDIT (Day 9) ------ banner "PERSISTENCE AUDIT" echo "--- Per-user crontabs ---" for user in $(cut -d: -f1 /etc/passwd); do cron=$(crontab -l -u "$user" 2>/dev/null) [[ -n "$cron" ]] && echo "=== $user ===" && echo "$cron" done echo "" echo "--- System cron locations ---" cat /etc/crontab 2>/dev/null ls -la /etc/cron.d/ 2>/dev/null ls -la /etc/cron.hourly/ /etc/cron.daily/ 2>/dev/null echo "" echo "--- Custom systemd services (potential persistence) ---" ls -la /etc/systemd/system/*.service 2>/dev/null | while read line; do warn "$line" done echo "" echo "--- Recently modified unit files ---" find /etc/systemd/system/ -name "*.service" -newer /etc/passwd 2>/dev/null | while read f; do alert "NEW SERVICE FILE: $f" grep "ExecStart" "$f" 2>/dev/null done echo "" echo "--- Services with suspicious ExecStart paths ---" grep -r "ExecStart" /etc/systemd/system/ 2>/dev/null \ | grep -iE "/tmp|/dev/shm|curl|wget|bash -c|python|perl|nc " | while read line; do alert "$line" done # ------ SECTION 3: NETWORK AUDIT (Day 10) ------ banner "NETWORK AUDIT" echo "--- Network interfaces ---" ip addr show echo "" echo "--- Listening ports with process owners ---" ss -tulnp echo "" echo "--- Active established connections ---" ss -tan state established | awk 'NR>1 {print $5}' \ | grep -vE "^127\.|^::1|^0\." | sort | uniq -c | sort -rn | head -20 echo "" echo "--- Routing table ---" ip route echo "" echo "--- ARP neighbours ---" ip neigh # ------ SECTION 4: SSH AUDIT (Day 11) ------ banner "SSH AUDIT" echo "--- Active SSH daemon settings ---" sudo sshd -T 2>/dev/null | grep -E "^permitrootlogin|^passwordauthentication|^pubkeyauthentication|^maxauthtries|^logingracetime|^port|^allowusers" echo "" echo "--- PasswordAuthentication check ---" if sudo sshd -T 2>/dev/null | grep -q "passwordauthentication yes"; then alert "PasswordAuthentication is ENABLED — key-only auth not enforced" else ok "PasswordAuthentication disabled" fi if sudo sshd -T 2>/dev/null | grep -q "permitrootlogin yes"; then alert "PermitRootLogin is ENABLED — direct root SSH allowed" else ok "PermitRootLogin disabled" fi echo "" echo "--- All authorized_keys files ---" find /home /root -name "authorized_keys" 2>/dev/null | while read f; do echo "[$f]" cat "$f" 2>/dev/null done echo "" echo "--- Recent successful SSH logins ---" grep "Accepted" /var/log/auth.log 2>/dev/null | tail -10 echo "" echo "--- Recent failed SSH attempts (top 10 IPs) ---" grep "Failed password" /var/log/auth.log 2>/dev/null \ | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 # ------ SECTION 5: PACKAGE AUDIT (Day 12) ------ banner "PACKAGE AUDIT" echo "--- Third-party repositories ---" cat /etc/apt/sources.list.d/* 2>/dev/null | grep -v "^#" | grep -v "^$" echo "" echo "--- Packages with security updates available ---" apt-get --just-print upgrade 2>&1 | grep "^Inst" | grep -i security | head -20 echo "" echo "--- Recently installed packages (last 7 days) ---" grep "install" /var/log/dpkg.log 2>/dev/null \ | awk -v d="$(date -d '7 days ago' +%Y-%m-%d)" '$1 >= d' | tail -20 echo "" echo "--- Unpackaged executables in /usr/local ---" find /usr/local/bin /usr/local/sbin -type f -executable 2>/dev/null | while read f; do warn "UNPACKAGED: $f" done # ------ SECTION 6: LOG SUMMARY (Day 13) ------ banner "LOG SUMMARY" echo "--- Auth log size ---" ls -lh /var/log/auth.log 2>/dev/null echo "" echo "--- Authentication statistics ---" echo "Failed login attempts : $(grep -c 'Failed password' /var/log/auth.log 2>/dev/null)" echo "Successful logins : $(grep -c 'Accepted' /var/log/auth.log 2>/dev/null)" echo "" echo "--- Top brute-force source IPs ---" grep "Failed password" /var/log/auth.log 2>/dev/null \ | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 echo "" echo "--- Recent sudo command history ---" grep "COMMAND=" /var/log/auth.log 2>/dev/null | tail -10 echo "" echo "--- Active auditd rules ---" sudo auditctl -l 2>/dev/null || warn "auditd not running or not installed" # ------ SECTION 7: SECURITY TOOLS CHECK ------ banner "SECURITY TOOLS STATUS" for tool in fail2ban auditd ufw aide lynis; do if systemctl is-active --quiet "$tool" 2>/dev/null; then ok "$tool is running" else warn "$tool is NOT running" fi done echo "" echo "--- fail2ban SSH ban status ---" sudo fail2ban-client status sshd 2>/dev/null || warn "fail2ban not configured for SSH" # ------ SUMMARY ------ banner "SCAN COMPLETE" echo "Report saved: $OUTPUT" echo "Scan time: $(date)" echo "" echo "Priority follow-ups:" echo " 1. Investigate any DELETED BINARY or SUSPICIOUS LOCATION process alerts" echo " 2. Review all NEW SERVICE FILE entries against known deployments" echo " 3. Verify any third-party apt repositories are legitimate" echo " 4. Check brute-force IPs — block the top offenders via ufw or fail2ban" echo " 5. Review recently installed packages for anything unexpected"

Red Team Perspective
How Week 2 skills chain in a real engagement

The complete post-exploitation workflow — Weeks 1 + 2 combined

A professional pentester's first 10 minutes after landing a shell follows a precise sequence. After two weeks of this roadmap, you understand every step:

1
Orient (Day 1–3): pwd, id, uname -a, ls -la. Know the layer, map the filesystem, find hidden files.
2
Local privilege escalation (Days 4–5): sudo -l, find / -perm -4000, GTFOBins. One misconfigured binary = root.
3
Process and service intel (Days 8–9): ps aux --forest reveals parent-child relationships, running services expose software versions for CVE lookup.
4
Network mapping (Day 10): ip route, ip neigh, cat /etc/hosts — build the internal network map without uploading any tools.
5
SSH key harvest + pivoting (Day 11): find / -name "id_rsa", cat ~/.ssh/known_hosts, reverse tunnel to C2. Lateral movement without uploading new tools.
6
Persistence (Days 9 + 13): Cron entry + systemd service + authorized_keys injection = three independent persistence mechanisms that survive reboots, session kills, and individual remediation attempts.

Your enum.sh script automates steps 3–5. Real engagement tools like LinPEAS cover the same ground but at greater depth — you now understand what every section of LinPEAS does because you built the concepts from scratch.

Blue Team Perspective
Week 2 hardening checklist — run on any new server

The complete Week 2 hardening checklist

Processes
Run ls -la /proc/*/exe | grep deleted weekly. Any result is an immediate investigation priority.
Services
Run find /etc/systemd/system/ -name "*.service" -newer /etc/passwd after any incident. Audit ExecStart paths for all custom units.
Persistence
Audit all cron locations quarterly. Every @reboot entry deserves explanation. User-level systemd timers in ~/.config/systemd/user/ require no root — check all users.
Network
Baseline ss -tulnp output. Any new listening port since last check is a finding. Enable ufw with deny-all-inbound as the default.
SSH
Disable PasswordAuthentication. Audit authorized_keys on every server quarterly. Install fail2ban. Run ssh-audit.com scan against your SSH server.
Packages
Enable unattended-upgrades for security patches. Audit /etc/apt/sources.list.d/ — every third-party repo is a trust decision. Run lynis monthly.
Logs
Deploy auditd rules from Day 13. Forward logs to a remote server via rsyslog. Run log-monitor.sh weekly via cron. Extend retention to 90 days minimum.

Pro Tips — Week 2 Mindset
Lessons from professionals who've built this foundation

The most important habit you built this week is thinking in layers. Every incident investigation starts with: what processes were running, what network connections were active, what services were configured, what logs show. That layered mental model — from process to network to service to log — is how experienced analysts triage in the first 10 minutes.

Your two scripts — recon.sh and enum.sh — are genuinely useful tools, not student exercises. Run them on any new Linux system you get access to: a VPS, a cloud instance, a CTF machine. The output will immediately give you situational awareness that would take 30 minutes of manual commands to build otherwise.

Set up a dedicated practice environment if you haven't already. A cheap VPS ($5/month on DigitalOcean or Vultr) gives you a real server to apply Week 2 hardening on, run your scripts against, and break deliberately to practice incident response. The cost is worthwhile — lab environments accelerate learning at 3× the speed of theory alone.

Common Week 2 mistake: skipping the HackTheBox Starting Point machines because they feel too simple. They are not too simple — they are correctly scoped for where you are now. "Meow", "Fawn", and "Dancing" each require exactly the network enumeration skills from Days 10–12. Completing them builds confidence and proves your skills work in a realistic environment.

Pro mindset going into Week 3: you now understand the server — its processes, services, network connections, and logs. Week 3 moves to the application layer — specifically HTTP and web applications. The same first-principles thinking applies: understand the technology, understand where it fails, understand what the logs show. The methodology doesn't change; only the target layer changes.
