So chmod 755 file means owner=rwx(7), group=r-x(5), others=r-x(5). This is the most common permission for executables.

Common permission patterns

755 (rwxr-xr-x)
Standard for executables and directories. Owner can do everything; others can read and execute. Default for most system binaries.
644 (rw-r--r--)
Standard for regular files. Owner reads and writes; others only read. Default for config files and documents.
600 (rw-------)
Private files. Owner only. Used for SSH private keys — anything else is a security misconfiguration.
777 (rwxrwxrwx)
Everyone has full access. Almost always a misconfiguration. Never use on production systems.
640 (rw-r-----)
Owner reads/writes, group reads, others have no access. Used for sensitive group-shared files like /etc/shadow.

SUID, SGID, and Sticky bit — the special bits

Beyond the 9 standard bits there are 3 special bits. These are where Linux privilege escalation vulnerabilities live.

SUID — Set User ID (bit 4 of the leading octal)
When set on an executable, it runs as the file's owner rather than the person executing it. Classic example: /usr/bin/passwd is SUID root — it needs root to write to /etc/shadow, but any user can run it. In ls -la you see s in the owner-execute position: rwsr-xr-x. If a vulnerable SUID binary exists, an attacker can use it to run commands as root.
SGID — Set Group ID (bit 2 of the leading octal)
On executables: runs as the file's group. On directories: new files created inside inherit the directory's group rather than the creator's group. Used for shared team directories. Shows as s in the group-execute position: rwxr-sr-x.
Sticky bit (bit 1 of the leading octal)
On directories (especially /tmp): users can only delete their own files, not files owned by others — even if the directory is world-writable. Shows as t in the others-execute position: rwxrwxrwt. Run ls -la / and look at tmp.

2. Practical Section
chmod · chown · find SUID · umask

Exercise 1 — read permissions like a pro

ls -la /etc/passwd # rw-r--r-- (644) — world-readable ls -la /etc/shadow # rw-r----- (640) — root + shadow group only ls -la /bin/passwd # rwsr-xr-x — SUID root (the s is the key) ls -la /tmp # rwxrwxrwt — world-writable + sticky bit ls -la /etc/sudoers # r--r----- (440) — root read-only stat /etc/passwd # detailed: permissions, owner, timestamps

Exercise 2 — chmod with octal notation

touch ~/security-lab/test-permissions.txt ls -la ~/security-lab/test-permissions.txt # default: 644 chmod 600 ~/security-lab/test-permissions.txt # private — owner only ls -la ~/security-lab/test-permissions.txt # now: rw------- chmod 755 ~/security-lab/test-permissions.txt # executable ls -la ~/security-lab/test-permissions.txt # now: rwxr-xr-x chmod 777 ~/security-lab/test-permissions.txt # DANGEROUS — for demo only ls -la ~/security-lab/test-permissions.txt # now: rwxrwxrwx chmod 644 ~/security-lab/test-permissions.txt # restore safe default

Exercise 3 — chmod with symbolic notation

chmod u+x script.sh # add execute for owner (u=user/owner) chmod g-w file.txt # remove write from group chmod o-rwx sensitive.txt # remove all permissions for others chmod a+r public.txt # add read for all (a=all) chmod u=rw,g=r,o= file # set precise: owner rw, group r, others nothing

Exercise 4 — chown (change ownership)

touch ~/security-lab/owned.txt ls -la ~/security-lab/owned.txt # shows your username as owner sudo chown root ~/security-lab/owned.txt # change owner to root ls -la ~/security-lab/owned.txt # now owned by root sudo chown root:root ~/security-lab/owned.txt # owner AND group to root sudo chown $USER:$USER ~/security-lab/owned.txt # give it back to yourself sudo chown -R $USER ~/security-lab/ # -R = recursive for whole dir

Exercise 5 — find all SUID binaries (critical recon command)

find / -perm -4000 -type f 2>/dev/null # all SUID files find / -perm -2000 -type f 2>/dev/null # all SGID files find / -perm -4000 -type f 2>/dev/null | xargs ls -la # detailed view find / -perm /6000 -type f 2>/dev/null # SUID or SGID (combined)

Exercise 6 — umask (default permission control)

umask # view current umask (typically 022) touch ~/test-umask.txt ls -la ~/test-umask.txt # 644 = 666 - 022 (umask subtracted) umask 077 # more restrictive: new files will be 600 touch ~/test-umask2.txt ls -la ~/test-umask2.txt # now 600 — private by default umask 022 # restore default

Exercise 7 — set the sticky bit and SUID (understand, don't abuse)

mkdir ~/security-lab/shared-dir chmod 1777 ~/security-lab/shared-dir # world-writable + sticky bit ls -la ~/security-lab/ | grep shared-dir # see the 't' at end of permissions chmod 4755 ~/security-lab/test-suid # set SUID on a file (demo only) ls -la ~/security-lab/test-suid # see the 's' in owner execute position
3. Real-World Scenario
SUID misconfiguration leads to full root access

A developer installs a custom backup tool on a production server and accidentally leaves it SUID root. An attacker with low-privilege access discovers it and escalates to root in seconds.

Step 1 — initial access, low privilege

The attacker has a shell as www-data via a web vulnerability. They run id — they're not root. Not yet.

Step 2 — SUID hunting

They run find / -perm -4000 -type f 2>/dev/null. Among standard system files they spot /opt/backup/run-backup owned by root with the SUID bit set. This isn't a standard binary.

Step 3 — exploit the SUID binary

The binary calls cp without a full path — it uses the environment's PATH. The attacker creates a fake cp in /tmp that spawns a shell, adds /tmp to the front of PATH, and runs /opt/backup/run-backup. The SUID binary executes as root, calls the fake cp, and the attacker gets a root shell.

Step 4 — full system compromise

With root they read /etc/shadow, add a new root user, install a backdoor, and exfiltrate everything. The entire escalation took under 2 minutes from finding the SUID binary.

What should have happened

The backup tool should never have been SUID root. It should have run as a dedicated low-privilege service account. Regular audits of SUID binaries (find / -perm -4000) would have caught the misconfiguration before an attacker did.

4. Red Team Perspective
Permission-based privilege escalation

The privesc checklist — permissions edition

After getting a foothold, a pentester systematically checks permissions for escalation paths. This is one of the most common routes to root on CTFs and real engagements.

find / -perm -4000 -type f 2>/dev/null # SUID files find / -perm -2000 -type f 2>/dev/null # SGID files find / -writable -type f -not -path "*/proc/*" 2>/dev/null # writable files find / -writable -type d 2>/dev/null # writable directories ls -la /etc/crontab /etc/cron.d/* 2>/dev/null # writable cron configs? sudo -l # what can I run as root?

GTFOBins — the attacker's SUID reference

GTFOBins (gtfobins.github.io) is a curated list of Unix binaries that can be abused to escape restrictions or escalate privilege when given SUID or sudo permissions. Common targets:

  • find with SUID or sudo — can execute arbitrary commands via -exec
  • vim / nano with sudo — can open a shell from within the editor
  • python / perl with SUID — can spawn a privileged shell in one line
  • cp / mv with sudo — can overwrite /etc/passwd or /etc/sudoers
  • bash with SUID — bash -p opens an effective-root shell instantly
bash -p # if bash is SUID — instant root shell find . -exec /bin/sh \; -quit # if find is SUID or in sudo -l python3 -c "import os; os.setuid(0); os.system('/bin/bash')" # python SUID

World-writable file exploitation

find / -perm -0002 -type f 2>/dev/null # 777 files find / -perm -0002 -type f 2>/dev/null | xargs ls -la 2>/dev/null echo "malicious_command" >> /path/to/world-writable-script.sh

If a root-owned script is world-writable and runs on a schedule (cron), an attacker can append commands that execute as root.

5. Blue Team Perspective
Audit · Harden · Monitor permissions

Audit SUID/SGID binaries regularly

Establish a known-good baseline of SUID binaries and alert on any new additions. This is a standard hardening step on any Linux server.

find / -perm -4000 -type f 2>/dev/null | sort > /root/suid-baseline.txt find / -perm -2000 -type f 2>/dev/null | sort > /root/sgid-baseline.txt diff /root/suid-baseline.txt <(find / -perm -4000 -type f 2>/dev/null | sort)

Fix common permission misconfigurations

chmod 644 /etc/passwd # should be readable by all but only root-writable chmod 640 /etc/shadow # root read/write, shadow group read, others nothing chmod 440 /etc/sudoers # read-only for root and sudo group chmod 600 ~/.ssh/id_rsa # SSH private key — owner only, no exceptions chmod 644 ~/.ssh/id_rsa.pub # public key can be world-readable chmod 700 ~/.ssh # .ssh directory — owner only find /var/www -perm -0002 2>/dev/null # find world-writable web files (critical)

Remove unnecessary SUID bits

chmod u-s /path/to/binary # remove SUID from a specific binary chmod g-s /path/to/binary # remove SGID from a specific binary

Monitor with auditd

auditctl -a always,exit -F arch=b64 -S chmod -S chown -k perm_change auditctl -a always,exit -F arch=b64 -S execve -F euid=0 -k root_exec ausearch -k perm_change # see all permission change events ausearch -k root_exec # see all commands executed as root

Set a restrictive umask in /etc/profile

A system-wide umask 027 ensures new files are created as 640 (owner rw, group r, others nothing) rather than the default 644, reducing accidental world-readable file creation.

echo "umask 027" >> /etc/profile # apply to all login shells system-wide

10. Pro Tips & Mindset
What separates beginners from professionals

Memorize the octal table cold. In CTFs and real engagements you need to decode 4755 or 2644 instantly without reaching for a calculator. The 4-2-1 pattern (read-write-execute) is the only thing to remember — everything else follows from it.

When you see an s where you'd expect an x in ls -la output, stop and investigate. That is SUID or SGID. On a custom binary that shouldn't need elevated privileges, it's a red flag.

Bookmark gtfobins.github.io today. It is to Linux privesc what a dictionary is to a writer. During a CTF or engagement, check every SUID binary you find against it within seconds.

Common mistake: confusing file permissions with directory permissions. For a directory, r means list contents, w means create/delete files inside, and x means traverse (enter) the directory. You can have x without r — you can enter but not list.

Common mistake: thinking chmod 777 is "fine just for testing." Permissions set for testing almost always make it to production. Treat every permission decision as permanent from the start.

Common mistake: using chown without -R on directories. Setting ownership on a directory does not change the files inside it. Always check whether you need the recursive flag.

Pro mindset: approach permissions as the principle of least privilege in action. Every extra permission bit is an unnecessary risk. Before adding a permission, ask: what is the minimum access this user or process actually needs? Default to restricting, not permitting.
