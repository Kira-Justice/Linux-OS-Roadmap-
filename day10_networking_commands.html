1. Theory Section
OSI model · TCP/IP · IP addressing · DNS · ports

The OSI model — the networking map

Every network communication, every attack, and every defence operates at one or more OSI layers. Knowing which layer a tool or attack operates at tells you what data it can see and what it can manipulate.

7
Application
HTTP, DNS, FTP, SSH — what the user sees. Most attacks target this layer.
6
Presentation
Encryption (TLS/SSL), encoding, compression. Where HTTPS decryption happens.
5
Session
Manages connections and sessions between applications.
4
Transport
TCP (reliable) and UDP (fast, unreliable). Ports live here. ss and netstat operate at this layer.
3
Network
IP addressing and routing. ip route, ping, traceroute operate here.
2
Data Link
MAC addresses, Ethernet frames. ARP operates here. ip link, arp.
1
Physical
Cables, wireless signals, hardware. Beyond what Linux commands can directly control.

Key networking concepts — all required knowledge

IP Address
Unique network identifier. IPv4: 4 octets (192.168.1.1). IPv6: 128-bit hex. A machine can have multiple IPs across multiple interfaces.
Subnet / CIDR
/24 = 256 addresses. /16 = 65,536. /32 = single host. The mask defines which bits are network vs host.
Port
0–65535. Ports 1–1023 are privileged (need root to bind). 1024–49151 registered. 49152–65535 ephemeral (client-side connections).
TCP vs UDP
TCP: handshake, reliable, ordered (HTTP, SSH, FTP). UDP: fire-and-forget, fast (DNS, VoIP, gaming). Both use ports.
DNS
Translates domain names to IPs. A record = IPv4, AAAA = IPv6, MX = mail, CNAME = alias, NS = nameserver, TXT = verification/SPF.
ARP
Maps IP addresses to MAC addresses on local networks. ARP poisoning/spoofing is a Layer 2 attack used for MITM.

2. Practical Section
ip · ss · netstat · dig · ping · traceroute · curl

Exercise 1 — ip: the modern interface tool

ip addr # all interfaces + IP addresses ip addr show eth0 # specific interface only ip -4 addr # IPv4 addresses only ip -6 addr # IPv6 addresses only ip link # link layer info (MAC, state UP/DOWN) ip link show eth0 # specific interface link info ip route # routing table (where does traffic go?) ip route show default # default gateway only ip neigh # ARP table: IP → MAC mappings on local network ip -s link # interface statistics: bytes in/out, errors

Exercise 2 — ss: socket statistics (modern netstat)

ss -tuln # TCP+UDP, listening, numeric (no DNS lookup) ss -tulnp # same + show process name and PID ss -tan # all TCP connections, numeric ss -s # summary statistics (established, closed, etc) ss -tp # TCP connections with process info ss -tulnp | grep :80 # who is listening on port 80? ss -tulnp | grep :443 # who is listening on HTTPS? ss state established # only established connections ss dst 8.8.8.8 # connections to a specific destination IP

Exercise 3 — netstat: the classic (still useful)

netstat -tuln # listening ports (similar to ss) netstat -tulnp # with process names (needs sudo for full info) netstat -an | grep ESTABLISHED # all established connections netstat -an | grep LISTEN # all listening ports netstat -rn # routing table (same as ip route) netstat -i # interface statistics netstat -s # protocol statistics (TCP/UDP/ICMP)

Exercise 4 — dig: DNS interrogation tool

dig google.com # A record (IPv4) for google.com dig google.com A # explicit A record query dig google.com AAAA # IPv6 address record dig google.com MX # mail server records dig google.com NS # nameserver records dig google.com TXT # text records (SPF, DMARC, verification) dig google.com ANY # all record types dig @8.8.8.8 google.com # query using specific DNS server (Google's) dig +short google.com # output IP only, no extra info dig -x 8.8.8.8 # reverse DNS: IP to hostname

Exercise 5 — ping, traceroute, and path analysis

ping google.com # basic reachability + latency ping -c 4 google.com # send exactly 4 packets then stop ping -i 0.2 google.com # fast ping every 0.2 seconds ping6 google.com # IPv6 ping traceroute google.com # hop-by-hop path to destination traceroute -n google.com # numeric only (no DNS lookups, faster) tracepath google.com # similar, no root needed mtr google.com # real-time combined ping+traceroute (install: apt install mtr)

Exercise 6 — curl and wget for HTTP investigation

curl https://example.com # GET request, print body curl -I https://example.com # HEAD only — response headers curl -v https://example.com # verbose: show full request+response curl -s https://ifconfig.me # your public IP address curl -L https://example.com # follow redirects curl -u user:pass https://api.example.com # basic auth curl -H "Authorization: Bearer token" URL # custom header curl -o /tmp/file.zip https://example.com/file.zip # download to file wget -q -O /dev/null -S https://example.com # check headers via wget

Exercise 7 — ARP and local network mapping

arp -n # ARP table (IP → MAC) ip neigh show # same via ip command cat /proc/net/arp # raw ARP table from kernel ip neigh | awk '{print $1,$5}' # just IP and MAC pairs
3. Real-World Scenario
Detecting data exfiltration via network command analysis

A SOC alert fires: unusually high outbound traffic from a production database server. The on-call analyst connects and begins triage using only the networking commands from today.

Step 1 — what connections are active right now?

The analyst runs ss -tulnp and spots an established connection on an unusual port — TCP to an external IP on port 4444. Port 4444 is a classic default for reverse shells and C2 frameworks. The connection is owned by python3.

Step 2 — identify the destination

dig -x [external-IP] returns no PTR record — the IP has no reverse DNS, consistent with a VPS used for malicious purposes. curl -s https://ipinfo.io/[external-IP] reveals it's registered to a cloud provider in an unexpected country.

Step 3 — trace the traffic volume

ip -s link show eth0 shows TX bytes far exceeding RX — much more data is going out than coming in. Combined with the established connection to an external IP, this confirms active data exfiltration in progress.

Step 4 — cross-reference with process and persistence

The analyst combines Day 8 and Day 9 skills: ps aux | grep python3 reveals the PID, /proc/[PID]/exe shows it came from /var/tmp/.update, and crontab -l -u postgres reveals a persistence mechanism launching it every 10 minutes.

Resolution

Block the destination IP at the firewall immediately, kill the process, remove the cron entry, investigate the initial access vector. The entire triage — from alert to containment decision — used only built-in Linux networking commands.

4. Red Team Perspective
Network reconnaissance · Pivoting · C2 detection evasion

Internal network mapping after foothold

Once inside a machine, a pentester maps the internal network to find other targets — without uploading any tools.

ip addr # what networks am I connected to? ip route # what subnets are routable from here? cat /etc/hosts # internal hostnames — reveals other servers cat /etc/resolv.conf # internal DNS server IP ip neigh # who else is on this local segment (ARP table) for i in $(seq 1 254); do ping -c1 -W1 192.168.1.$i &>/dev/null && echo "192.168.1.$i is up" done # subnet sweep using only ping

Identifying outbound egress paths

curl -s https://ifconfig.me # what's the public IP? (confirms outbound HTTP works) curl -s http://portquiz.net:80 # test outbound TCP 80 curl -s http://portquiz.net:443 # test outbound TCP 443 curl -s http://portquiz.net:8080 # test outbound TCP 8080 dig @8.8.8.8 google.com +short # confirms outbound DNS (UDP 53) works

Knowing which ports can reach out determines which protocol to use for a reverse shell or C2 channel. If only 443 is allowed outbound, the attacker uses HTTPS-based C2.

DNS as an intelligence source

dig target.com ANY # all DNS records in one query dig target.com MX # mail servers dig target.com TXT # SPF records reveal email infrastructure dig target.com NS # nameservers (registrar info) host -t AXFR target.com ns1.target.com # zone transfer attempt (often disabled but worth trying) dig +short target.com # clean IP list for scripting

Staying quiet — avoiding noisy network behaviour

  • Use ss and ip instead of running external scanners — they're built-in and leave no new process traces
  • Slow down reconnaissance — rapid ping sweeps trigger IDS alerts
  • Use DNS over HTTPS or trusted resolvers to blend DNS queries into normal traffic
  • Route C2 traffic over port 443 to blend with HTTPS — the most consistently allowed outbound port
5. Blue Team Perspective
Network baselining · Anomaly detection · Firewall hardening

Establish a network baseline

ss -tulnp > /root/network-baseline.txt # save clean listening port baseline diff /root/network-baseline.txt <(ss -tulnp) # compare current vs baseline ip route > /root/route-baseline.txt # save routing baseline

Detect anomalous outbound connections

ss -tan state established | awk 'NR>1{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn # which external IPs have the most established connections? ss -tan | grep -vE "127\.|::1|0\.0\.0\.0" | awk '{print $5}' | grep -oE "[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+" | sort | uniq # all unique external IPs currently connected

Firewall hardening with ufw and iptables

sudo ufw enable # enable firewall sudo ufw default deny incoming # block all inbound by default sudo ufw default allow outgoing # allow all outbound (tighten later) sudo ufw allow 22/tcp # allow SSH sudo ufw allow 80/tcp # allow HTTP sudo ufw allow 443/tcp # allow HTTPS sudo ufw deny 23/tcp # block Telnet explicitly sudo ufw status verbose # review all rules sudo ufw logging on # enable firewall logging

Monitor DNS for suspicious lookups

dig -x [suspicious-IP] # reverse DNS — does it have a legitimate name? cat /var/log/syslog | grep "named\|resolve" # local DNS queries in syslog ss -unlp | grep :53 # what is listening on DNS port locally?

Add network audit to recon.sh

banner "NETWORK AUDIT" echo "--- Listening ports ---" ss -tulnp echo "--- Established external connections ---" ss -tan state established | awk 'NR>1{print $5}' \ | grep -vE "^127\.|^::1|^0\." | sort | uniq -c | sort -rn echo "--- Routing table ---" ip route echo "--- ARP neighbours ---" ip neigh

7. Daily Tasks & Challenges
Complete before moving to Day 11
1
Run ss -tulnp on your system and map every listening port to its service. For each port, answer: what service is it, does it need to be exposed, and what would an attacker learn from it? Save this as your network attack-surface map in ~/security-lab/notes/day10-network-surface.txt.
2
Use dig to query 3 different real domains and collect every record type — A, MX, NS, TXT. For each domain explain what the TXT records tell you (SPF policy, DMARC, any verification tokens). This is exactly what an attacker does during external DNS reconnaissance.
3
Use curl -I to retrieve response headers from 5 different websites. Note the Server header (reveals web server software + version), X-Powered-By (reveals backend stack), and any security headers like Strict-Transport-Security or Content-Security-Policy. 10. Pro Tips & Mindset
How professionals think about networking

Use ss instead of netstat on modern systems. netstat is from the old net-tools package — deprecated, often not installed by default, and slower. ss reads directly from the kernel socket tables and is faster, more informative, and always available. Learn both because you'll encounter both in the real world, but default to ss.

Always use -n (numeric) flags with ss, netstat, and traceroute unless you specifically want hostname resolution. DNS lookups for every IP address slow these commands dramatically on systems with poor DNS connectivity — on a compromised server during an incident, that delay could be costly.

dig is vastly more powerful than nslookup. Learn dig. The +short flag makes it scriptable — dig +short target.com returns just the IP with no noise, perfect for piping into other commands or scripts.

Common mistake: thinking open ports equal vulnerabilities. An open port is only a vulnerability if the service behind it has a weakness or is misconfigured. A properly patched, correctly configured SSH on port 22 is not a vulnerability — it is a managed access point. The question is always: does this port need to be open, to whom, and is the service behind it secure?

Common mistake: using ping alone to confirm a host is down. Many firewalls and servers block ICMP. A host that doesn't respond to ping may still have open TCP ports. Use curl or ss to confirm actual service availability, not just ICMP reachability.

Pro mindset: think of the network layer as a second attack surface map that exists independently from the filesystem. Days 1–7 mapped the local attack surface (files, permissions, users). Days 8–10 are mapping the runtime attack surface (processes, services, network connections). A complete security picture requires both — and recon.sh is growing to capture both in one tool.
